# docs-sync

여러 레포를 **git pull** 하고, 각 레포의 **pull 전후 diff를 근거로 대응 문서를 자동 갱신**하는 스킬. 사용자가 선택한 동작은 **pull까지 수행 → 자동 반영 → 요약 보고**다.

> 핵심 원칙
> - **사실 기반**: 문서에 쓰는 모든 항목은 실제 diff(커밋 제목·변경 파일·코드)에서 확인한 것만. 추측·일반론 금지. 불명확하면 그 부분은 "미확인"으로 남긴다.
> - **비파괴**: 문서 전체를 다시 쓰지 않는다. 그 문서의 기존 구조를 따라 **변경 로그/갱신일 섹션만** 갱신한다.
> - **문서 컨벤션 존중**: 각 문서가 이미 쓰는 형식(예: 상단 "master pull 보강" 블록, "최근 변경사항" 표, "분석 갱신일")을 그대로 따른다.

## 설정 (첫 실행 시)
작업 시작 전 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/mentis-docs/config.env`를 source 해 환경 변수를 불러온다. 파일이 없으면 그 절차대로 기본값을 확인·저장한 뒤 진행한다. 이하 경로는 `$DOCS_REPOS_ROOT`·`$DOCS_ROOT` 기준.

## 사용법
- `/docs-sync` — `$DOCS_REPOS_ROOT` 하위 모든 대상 레포를 pull·갱신.
- `/docs-sync {레포...}` — 지정한 레포만. 예: `/docs-sync wadiz-frontend app-api`.
- `/docs-sync pull=off` — pull은 건너뛰고, 각 레포의 최근 미반영 변경(`ORIG_HEAD..HEAD` 또는 최근 N개 커밋)만 문서화. 이미 pull 해둔 경우에 사용.

## 절차

### 1. 대상 레포 선정
`$DOCS_REPOS_ROOT`의 하위 항목 중 다음을 **제외**한다: 디렉터리가 아닌 것, `.git`이 없는(=git 저장소 아님) 폴더, `$DOCS_ROOT` 자신, `DOCS_SYNC_EXCLUDE` 목록. 인자로 레포를 지정했으면 그 교집합만.

### 2. 레포별 pull 전후 스냅샷 (핵심 — 정확한 diff 범위 확보)
각 레포에서 순서대로:

```bash
cd "$DOCS_REPOS_ROOT/{repo}"
BRANCH=$(git rev-parse --abbrev-ref HEAD)
BEFORE=$(git rev-parse HEAD)
# pull=off 가 아니면:
git pull --ff-only 2>&1 || git pull 2>&1   # ff 불가 시 일반 pull. 충돌/에러는 그대로 기록하고 그 레포는 갱신 건너뜀
AFTER=$(git rev-parse HEAD)
```

- `BEFORE == AFTER` → **변경 없음**, 그 레포는 스킵(요약에 "변경 없음"만).
- pull이 **충돌/실패**하면 문서를 건드리지 말고 요약에 오류로 보고(사용자가 직접 해결).
- `pull=off`면 `BEFORE`는 `git rev-parse ORIG_HEAD`(있으면) 또는 최근 커밋 기준으로 잡고, 없으면 그 레포는 "범위 불명"으로 보고.

> `execute_all git pull`을 한 번에 돌리지 않고 **레포별로 pull 전 HEAD를 기록**하는 이유: 정확한 `BEFORE..AFTER` 범위를 알아야 "이번 pull로 실제 들어온 변경"만 문서화할 수 있다.

### 3. 변경 내용 수집 (사실 근거)
변경이 있는 레포마다:

```bash
git log --no-merges --format='%h %s' "$BEFORE..$AFTER"     # 커밋 제목(이슈키 포함)
git diff --stat "$BEFORE..$AFTER"                           # 변경 파일 규모
```

- 커밋 제목에서 **이슈키**(예 `FE1-1234`, `QA-22345`, `CLIENT-101`)를 추출한다.
- 문서에 쓸 "무엇이 바뀌었나"가 제목만으로 불명확하면 **해당 파일의 실제 diff를 더 읽어** 사실을 확인한다(`git diff "$BEFORE..$AFTER" -- <경로>`). 추측 금지.
- (선택·있으면) 이슈키로 Jira 제목을 조회해 요약을 보강한다. Jira 접근이 없으면 커밋 제목만 쓴다.
- 변경 커밋이 많으면(예 30개 초과) 레포별로 서브 에이전트에 "이 범위 diff를 읽고 문서 갱신용 변경 테마 목록(테마·이슈키·대표 파일경로)을 뽑아라"라고 위임해도 된다.

### 4. 레포 → 문서 매핑
`reference/config.md`의 "레포 → 문서 매핑 규칙"을 따른다: `DOCS_SYNC_MAP` → `$DOCS_ROOT/{repo}.md` → `$DOCS_ROOT/{repo}/`(폴더면 대표 문서) 순. 없으면 **"문서 없음(신규 후보)"** 로 분류하고 새 문서를 만들지 않는다(요약에만).

- 폴더형 문서(예 `wadiz-frontend/`)는 변경 파일 경로를 보고 **가장 관련 있는 하위 문서**를 고른다. 대표/인덱스 문서에 변경 로그가 모여 있으면 거기에 쓴다. 애매하면 인덱스 문서에 기록한다.

### 5. 문서 갱신 (자동 반영 — 비파괴)
대상 문서를 **읽고 그 문서의 기존 형식을 파악한 뒤** 그 형식대로 갱신한다:

- **갱신일/버전 표기**가 있으면 오늘 날짜로 갱신한다(오늘 날짜는 대화 컨텍스트의 현재 날짜를 쓴다. 셸에서 날짜를 새로 만들지 않는다).
- **변경 로그 섹션**(예 "최근 변경사항" 표, 상단 "pull 보강" 블록)이 있으면 그 형식에 맞춰 이번 변경 테마를 **이슈키·대표 파일경로와 함께** 추가한다.
- 그런 섹션이 없으면 문서 맨 위(개요 다음)에 간결한 "최근 변경사항 (YYYY-MM-DD pull 기준)" 블록을 새로 만들되, **그 문서의 톤과 서식**(한국어/존댓말, 표/불릿 스타일)을 맞춘다.
- 본문의 서술이 이번 변경으로 **사실과 달라졌으면**(예: 삭제된 기능·바뀐 경로) 해당 서술만 정정한다. 관련 없는 부분은 건드리지 않는다.
- 커밋이 순수 리팩터/포맷/빌드설정처럼 **문서에 반영할 사용자 의미가 없으면** 문서를 바꾸지 않고 요약에만 "문서 반영 불필요"로 남긴다.

> 문서 저장소(`$DOCS_ROOT`)가 git 저장소면, 변경은 작업트리에만 남긴다. `DOCS_SYNC_BRANCH_PREFIX`가 설정돼 있을 때만 그 접두사로 브랜치를 만들어 커밋하고, **푸시는 사용자 승인 후에만** 한다(기본은 커밋도 하지 않음).

### 6. 요약 보고
마지막에 레포별 결과를 표로 보고한다:

| 레포 | 변경 커밋 | 이슈키 | 문서 | 처리 |
|---|---|---|---|---|
| wadiz-frontend | 12 | FE1-1234 외 | wadiz-frontend/… | 갱신함 |
| app-api | 3 | QA-22345 | app-api.md | 갱신함 |
| helm-charts | 5 | — | (문서 없음) | 신규 후보 |
| nicepay-api | 0 | — | nicepay-api.md | 변경 없음 |
| co.wadiz.fep | 2 | — | co.wadiz.fep.md | 문서 반영 불필요(리팩터) |

- 문서를 실제로 고친 파일 목록과, pull 충돌/실패 레포, "문서 없음(신규 후보)" 레포를 각각 명확히 나열한다.
- 문서 저장소에 커밋/푸시했는지 여부도 명시한다(기본: 안 함).

## 주의
- pull은 `--ff-only`를 먼저 시도한다. 로컬 커밋·충돌이 있는 레포는 **자동 병합하지 않고** 오류로 보고한다(사용자 데이터 보호).
- 문서 저장소가 아닌 원본 레포의 소스는 **절대 수정하지 않는다**. 이 스킬은 pull(읽기 갱신)과 문서(docs) 수정만 한다.
- 오늘 날짜는 대화의 현재 날짜를 사용한다(스킬 실행 환경에 따라 셸 날짜 생성이 제한될 수 있다).
