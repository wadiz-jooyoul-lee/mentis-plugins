# 분석·구현 원칙 (사실 기반 · 활성 경로 · 반증)

go-dobby의 모든 **분석(dobby-start)·구현(dobby-impl)·산출/조사(dobby-produce·Explore)** 에이전트가 지켜야 하는 공통 원칙. 스폰되는 에이전트 프롬프트 **상단에 1·2·3을 기본 삽입**하고, 국내/글로벌 이중 경로가 얽힌 작업이면 **1·4를 반드시 포함**한다.

## 1. 활성 경로 확정 먼저 (가장 중요)

분석·수정 전에, 이 기능이 **실제로 실행되는 코드 경로**를 먼저 확정하라.
- 이 저장소군은 국내 기능이 `static/entries/*`(레거시)와 `apps/global`(이관본, `korea-routes/`·`routes/`)로 **이중화**되어 있고, 동일 기능 코드가 양쪽에 중복 존재한다.
- 특정 URL(예: `/web/campaign/detail/:id`)을 "어느 앱/엔트리/라우트가 실제로 렌더하는지"를 **라우트 등록·엔트리 빌드·마운트 지점으로 증명**한 뒤에만 그 코드를 근거로 삼아라.
- 코드가 "존재"하는 것과 "실행"되는 것은 다르다. 실행되지 않는 쪽은 **데드 코드로 명시하고 분석에서 제외**하라.
- 활성 경로를 확정하지 못하면 **거기서 멈추고 보고**하라. 추측으로 한쪽을 고르지 마라.

## 2. 전제를 검증·반증하게 만들기 (확증 편향 차단)

전달받은 가설/전제는 **"검증 대상"이지 "사실"이 아니다.**
- 먼저 이 전제가 실제 활성 코드에서 성립하는지 **독립적으로 검증**하라.
- 성립하지 않으면 전제를 **반박**하고, 실제 코드가 말하는 다른 설명을 제시하라.
- 주어진 프레임에 맞추려 하지 말고, **코드가 가리키는 결론을 우선**하라.

## 3. 사실/추측 구분 + 미증명 시 중단 (원칙 위반 차단)

모든 결론은 **실제 코드를 직접 읽어 확인한 사실**이어야 하며 **`파일:라인`으로 근거**를 대라.
- 정적 코드로 증명 가능한 것과 불가능한 것(런타임 타이밍, 네이티브 앱 등 저장소 밖)을 명확히 **구분해 표기**하라.
- 증명하지 못한 메커니즘 위에 **수정 설계를 세우지 마라.** 미증명이면 **"미증명 + 확인 방법(재현/디바이스)"으로 보고하고 멈춰라.**

## 4. 도달 가능성 증명 (데드코드 오판 차단)

어떤 코드를 원인/수정 대상으로 지목할 때는 **"그 코드가 실제로 도달·실행되는가"**를 증명하라.
- 라우트/포털이 실제 마운트되는지, 해당 핸들러가 실제 트리거되는지(클릭·이벤트·URL 매칭 패턴)를 **`파일:라인`으로 확인**.
- URL 패턴 매칭이 관건이면, **실제 URL 문자열을 그 정규식/패턴에 대입해 매칭 여부를 명시**하라.

---

**넣는 위치**
- 분석(dobby-start)·구현(dobby-impl)·조사/산출(dobby-produce·Explore) 에이전트 프롬프트 **상단에 1·2·3** 기본 삽입.
- **국내/글로벌 이중 경로**가 얽힌 작업이면 **1·4를 반드시 포함**.
- 오케스트레이터(dobby-order)는 **P1 분석 전에 "활성 경로 확정" 게이트**를 두고, 확정 전까지 **P4 구현으로 넘어가지 않는다.**
