# dobby-order

모든 작업의 **유일한 진입점**이자 오케스트레이터다. 이 스킬을 실행하는 주체가 **메인 에이전트**다. 진입한 작업을 분석해 필요한 에이전트 수(**팬아웃 K**)를 정하고, 서브 에이전트로 착수·구현시키고, 리뷰 루프를 거쳐 상위 브랜치로 통합한다.

> 단독 이슈 해결을 위한 별도 경로는 없다. **단일 버그는 그냥 "에이전트 1명(K=1)"인 오케스트레이션**이다 — 특수 경우가 아니라 팬아웃 수가 1일 뿐이며, 메타·계약·리뷰 등 절차와 산출물은 K와 무관하게 **동일**하다(에이전트·계약·보드 행이 1개가 될 뿐).

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/go-dobby/config.env`를 source 해 환경 변수를 불러오고, **이미 값이 있는 변수는 묻지 않고** 빠진 값만 규칙대로 채워 저장·export 한다. 메타 루트 `ORCHESTRATION_META`, 변수 목록·기본값, 폴더 배치(워크트리 `$ORCHESTRATION_WORKSPACE/subtree/` · 메타 `$ORCHESTRATION_META/`)는 모두 그 문서에 있다. 이하 메타 경로는 `$ORCHESTRATION_META` 기준.

## 스킬 관계 (building block)

```
dobby-order   ← 오케스트레이터(메인). 전체 흐름 지휘.
  ├─ dobby-start    ← 에이전트별 착수·분석. 메인이 P1에서 실행.
  ├─ dobby-impl     ← 에이전트별 구현(work-type=code). 서브 에이전트가 P4에서 실행.
  └─ dobby-produce  ← 에이전트별 산출(work-type=비소스: 문서·리서치 등). P4 대안.
```

리뷰·검증·정리:
- **code 작업**: `dobby-test`(브라우저 검증) → `dobby-resolve`(해결 표시) → `dobby-end`(정리).
- **비소스 작업**: 브라우저 테스트 없음 — P5 내용 리뷰로 검증하고 → `dobby-resolve` → `dobby-end`(meta 산출이면 워크트리 정리 no-op).
- 이들 중 dobby-test·resolve·end는 통합 이후 사용자가 실행한다.

## 입력 / 진입 모드

`/dobby-order {대상} [base=] [subtasks=] [agents=] [mode=]`

- **(a) 이슈 키/URL**: 예 `FE1-1187` 또는 `$JIRA_BASE_URL/browse/FE1-1187`. 이슈 안의 기획·피그마·디자인을 확인해 진행.
- **(b) 이슈 + 기획서/피그마**: 이슈와 함께 문서·피그마 링크를 받아 진행.
- **(c) 문서 전용(이슈 없음)**: 특정 문서만 주고 작업을 요청. 이슈 키가 없으므로 **작업 키를 생성**한다.
  - 규칙: **`TASK-{slug}`** — `slug`는 문서 제목/주제에서 뽑은 짧은 영문 kebab. 충돌 시 `-YYYYMMDD` 부가(예: `TASK-login-refactor`).
  - 이 키가 이슈 키 자리를 그대로 대체한다: 폴더 `$ORCHESTRATION_META/TASK-{slug}/`, 브랜치 `feature/TASK-{slug}`.
  - **Jira 조회·상태 전환은 건너뛴다.** 생성한 작업 키는 시작 계획 게이트(아래 "확인 게이트" #1)에서 사용자에게 1회 확인한다.

인자:
- `base=`: 상위 브랜치의 베이스(미지정 시 `$ORCHESTRATION_DEFAULT_BASE`, 기본값 master).
- `subtasks=키,키`: 대상 하위이슈 명시(일부만). `agents=N`: 팬아웃 강제. 미지정 시 **자동 판단**.
- `mode=A|B`: 실행 모드(미지정 시 A).

## 팬아웃 K 결정 (진입 판단)

이슈 타입이 아니라 **코드 범위(파일 오너십)**로 판단한다. 타입·하위이슈는 강한 힌트일 뿐이다.

1. 이슈/문서 조회 + 하위이슈 탐색(`parent = {상위}` JQL) + (멀티 repo면) repo별 **읽기 전용 Explore 에이전트**로 범위를 사실 기준으로 분석한다.
2. 분리 가능한 오너십 영역을 도출한다:
   - 병렬 가능한 영역이 **2개 이상** → `K = 영역 수`(다중). 영역과 하위이슈를 **1:1**로 맞춘다(없으면 그 분해로 하위이슈 생성).
   - **한 덩어리(cohesive)** → `K=1`(단일). 하위이슈 불필요.
   - **멀티 repo → repo당 최소 1 에이전트**(repo 단위 오너십 = 통합 시 머지 충돌 원천 차단).
3. 판단에 필요한 사실이 불확실하면 **묻지 말고 백그라운드 에이전트로 직접 확인**한다(추측 금지).

### 브랜치 토폴로지 (하나의 규칙 — K로 분기하지 않는다)

- **루트 브랜치** `{prefix}/{루트키}`를 `{base}`에서 만든다. 통합 지점이다.
- **에이전트 브랜치**: 에이전트키 ≠ 루트키인 에이전트는 `{prefix}/{에이전트키}`를 루트 브랜치에서 만들어 작업하고, 리뷰 통과 후 **루트 브랜치로 머지**한다.
- 에이전트가 하나뿐이면 그 에이전트키 = 루트키이므로 **루트 브랜치가 곧 에이전트 브랜치**다 — 추가 브랜치도, 통합 머지도 자연히 없다(별도 부모 브랜치를 만들지 않는다). 이는 "에이전트 수 1"의 자연스러운 귀결일 뿐 특수 분기가 아니다.
- `{base} └ {prefix}/{루트키} [├ {prefix}/{에이전트A} └ {prefix}/{에이전트B} …]` (대괄호 부분은 에이전트가 여럿일 때 생긴다.)

## 절차 (Phase)

전이는 **객관적 조건**으로 자율 진행한다(매 단계 확인하지 않는다). 아래 "전이 조건표"·"확인 게이트" 참조.

### P0. 셋업
- 소스 루트에서 `git fetch origin {base}`.
- **루트 브랜치·워크트리 생성** + origin 푸시: `git worktree add -b {prefix}/{루트키} $ORCHESTRATION_WORKSPACE/subtree/{repo}-{루트키} origin/{base}`. (에이전트가 여럿이면 이 루트 브랜치에서 각 에이전트 브랜치가 갈라진다. 하나면 이 브랜치가 곧 그 에이전트의 작업 브랜치다.)
- **메타 루트는 항상 동일**: 루트 키 폴더 `$ORCHESTRATION_META/{루트키}/`에 `orchestration.md`·`agents/`·`reviews/`를 **항상** 둔다(K로 분기하지 않음 — 규격은 아래 "오케스트레이션 메타"). 하위 에이전트 키가 따로 있으면 그 키 폴더(`$ORCHESTRATION_META/{에이전트키}/`)에는 work 파일이 들어간다.
- **멀티 repo면 저장소마다 위를 반복.** base는 저장소마다 다를 수 있다(예: FE master, BE develop).

### P1. 착수·분석
- 각 대상(하위이슈 또는 단일 이슈/작업)에 `dobby-start {키} base={상위 브랜치 또는 base}` 실행 → 워크트리 생성·코드 분석·`analysis.md`·`test-plan.md` 확보.
- **하위이슈가 없거나 새로 만들 때**: 먼저 P3 범위 배분으로 **에이전트 단위 분해**한 뒤 1:1 하위이슈를 생성하고 각각 dobby-start를 실행한다(이 경우 P3가 P1보다 먼저).
- 착수 전 각 저장소 원격에서 상위이슈 관련 프로토타입/스파이크 브랜치를 검색해, 재사용 가능분만 **사실 기준**으로 계약에 반영한다.

### P2. 분석 검증
- 분석 산출물을 숙지하고 **기획서·Jira·(있으면) Figma/문서**와 교차 검증한다. 어긋나면 해당 dobby-start 분석을 보정한다.
- 미결 결정이 있으면 목록화한다. **코드로 확인 가능한 건 확인해 정하고**(백그라운드 에이전트), 순수 제품/디자인 결정만 사용자에게 승인받는다. 미결을 남긴 채 P3로 넘어가지 않는다.

### P3. 범위 배분 + work-type 분류
- **구현에 필요한 코드를 먼저 분석**해 에이전트별 **수정 파일 화이트리스트**를 정한다.
- 오너십이 안 겹치면 → **병렬**로 바로 P4(사용자에게 묻지 않음). 겹치면 → **순차**로 자동 결정(사유 기록).
- 멀티 repo면 각 에이전트는 **한 저장소에만** 속하게 배분하고, 저장소 간 결합점은 **교차 저장소 API 계약** 하나로 모아 먼저 확정한다.
- **work-type 분류(라우팅)**: 각 에이전트 작업을 분류한다.
  - `work-type = code` → 구현은 `dobby-impl`, 검증은 `dobby-test`.
  - `work-type = 비소스`(문서·리서치·분석·설정 등) → 산출은 `dobby-produce`, 검증은 **P5 내용 리뷰**(브라우저 테스트 없음). 산출물이 repo면 자기 브랜치 푸시, repo 밖이면 `$ORCHESTRATION_META/{키}/deliverables/`에 저장.
  - 한 오케스트레이션에 두 work-type이 섞일 수 있다(에이전트별로 다르게 라우팅).
- **`orchestration.md`·`agents/{슬러그}.md`(에이전트별 계약)·`agents/review-agent.md`를 항상 작성한다**(K로 분기하지 않음). 계약에 수정 허용 범위(화이트리스트)·공유 데이터 계약을 명시한다. **에이전트가 하나뿐이면 계약도 1개**이며, 그 화이트리스트는 **전체 범위**(오너십 제약 없음)로 적는다. 보드(orchestration.md)는 에이전트 1행이 된다.

### P4. 구현
- 각 서브 에이전트를 기동한다(모드 A/B, 아래 "실행 모드"). 에이전트가 하나뿐이어도 **반드시 서브 에이전트로 기동한다**(메인이 직접 구현하지 않는다 — 리뷰 분리·경로 단일화). **계약 파일 경로(`agents/{슬러그}.md`)를 첫 지시로 전달한다**(에이전트 수와 무관하게 항상 계약을 전달).
- work-type으로 실행 스킬을 정한다:
  - `code` → `dobby-impl`로 구현 → `status.md` `구현중`·`implementation.md` 기록 → `commit --no-verify` → 자기 브랜치 푸시.
  - `비소스` → `dobby-produce`로 산출 → `산출중`·`produce.md` 기록 → (repo 산출이면) 자기 브랜치 푸시 / (meta 산출이면) `deliverables/` 저장.
- 어느 쪽이든 끝나면 메인에 리뷰 요청·대기.

### P5. 코드리뷰 (오케스트레이터 단일 리뷰)
- **별도 리뷰 에이전트**를 기동해 각 브랜치 diff를 **적대적으로** 리뷰한다(브랜치별 병렬 가능). **K=1이어도 동일하게 별도 리뷰 에이전트가 리뷰한다**(리뷰어≠구현자 — 구현자 dobby-impl은 셀프리뷰하지 않으므로 이 리뷰가 유일한 코드리뷰다). 저장소마다 그 repo 규약(계층형 CLAUDE.md·리뷰봇 규칙)으로 리뷰한다.
- 리뷰 대상 diff: `git diff {fork}..HEAD` (`fork` = `git merge-base origin/{base} HEAD`).
- **리뷰 루브릭**(리뷰 에이전트에 전달):
  - **A. 정확성/품질**: 버그·논리 오류, null/undefined·경계값, 비동기/에러 처리, 회귀(소비처 영향), 불필요한 복잡성·데드코드, 설계-구현 불일치.
  - **B. 규약 적합성**: 그 워크트리·영역의 CLAUDE.md/NAMING/GLOSSARY를 **직접 읽어** 대조.
  - **C. CI 고유 규칙**: `.github/workflows`에서 읽어 반영(예: `packages/i18n/**`는 키 rename/삭제만 지적).
  - **D. 커밋 메시지**: `fix:`/`feat: {키} ...`, 의미 단위 커밋.
  - **E. 범위 준수**: 각 에이전트가 계약 화이트리스트 범위만 수정했는지 검증(오케스트레이터 리뷰만 할 수 있는 확인). 에이전트가 하나뿐이면 화이트리스트가 전체 범위라 이 항목은 사실상 통과지만, 검사 절차 자체는 동일하게 수행한다.
  - 심각도: **blocker**(병합 불가) / **major**(강권) / **minor·nit**(사소).
- **비소스 작업(work-type=비소스)**은 위 코드 루브릭 대신 `dobby-produce`의 **"내용 리뷰 기준"**(정확성·완결성·요청 부합·근거 명시·규약)으로 리뷰한다.
- 결과를 **항상** `$ORCHESTRATION_META/{루트키}/reviews/round-{n}/{슬러그}.md`에 심각도·`파일:라인`·수정 제안으로 남긴다(에이전트 수와 무관).

### P6. 피드백 루프
- 리뷰 결과를 해당 서브 에이전트에 전달(모드 A는 `SendMessage`로 같은 에이전트 이어감). `dobby-impl` 피드백 반영으로 재푸시.
- **모든 수정 라운드 후 반드시 재리뷰(생략 금지).** blocking(blocker·major)=0이 될 때까지 `리뷰중`↔`수정중` 반복. **최대 라운드 기본 3**, 초과 시 사용자 에스컬레이션.

### P7. 통합
- 리뷰 클린 → 각 에이전트 브랜치를 **루트 브랜치로 순차 머지·푸시**한다(멀티 repo면 저장소별 루트 브랜치로 각각). 에이전트가 하나뿐이면 그 브랜치가 곧 루트 브랜치라 머지할 것이 없다(자기 브랜치가 결과물) — 이는 규칙의 자연스러운 귀결이며 별도 분기가 아니다.
- **⛔ `$ORCHESTRATION_DEFAULT_BASE`(정식 배포 베이스)로의 PR 생성·머지는 금지(금기).** 사용자가 요청해도 이 스킬 범위에선 하지 않고, 별도 절차로 사용자가 직접 진행하도록 안내만 한다.

## 전이 조건표

| 전이 | 판단 조건(객관적) | 확인 |
|------|-------------------|:----:|
| 진입 → 계획 확정 | 이슈/문서 + 하위이슈 탐색 + 범위 분석 | ✅ 초기 1회 |
| P0 → P1 | 브랜치·워크트리 생성됨 | 자동 |
| P1 → P2 | 모든 에이전트 `analysis.md` 산출 | 자동 |
| P2 → P3 | 분석이 기획·Jira/문서와 일치 | 자동(순수 제품/디자인 미결만 확인) |
| P3 → P4 | 파일 오너십 화이트리스트 확정 | 자동(겹치면 순차 자동 결정) |
| P4 → P5 | 에이전트 "구현·푸시 완료" 보고 | 자동 |
| P5 → P6 | 리뷰 findings 산출 | 자동 |
| P6 → 통합 | blocking=0 | 자동(최대 라운드 초과 시 확인) |
| P7 | 리뷰 클린 → 부모 브랜치 머지 | 자동(master는 금지) |

## 확인 게이트 (좁음 — 이때만 사용자를 부른다)

1. **초기 계획 1회**: 팬아웃 K, 하위이슈/작업 키(문서 전용의 `TASK-{slug}` 포함), 범위 배분.
2. **순수 제품/디자인 결정**: 코드로 확정 불가한 것.
3. **최대 리뷰 라운드 초과**(기본 3).
4. **코드 외 작업**: 인프라·외부 콘솔 설정·배포 순서.
5. **메인 브랜치 밖 반영**(배포 등). — master PR·머지는 확인이 아니라 **금지**.

> **게이트 전 필수 규칙(docs 참조 §강화):** 사용자 결정이 필요해 보이면 **묻기 전에 관련 코드를 다시 상세 분석**하고(필요하면 docs 참조 — `$ORCHESTRATION_DOCS_ROOT/{repo}` 문서 → 없으면 `$ORCHESTRATION_REPOS_ROOT/{repo}` 실제 소스), 스스로 판단 불가한지 재확인한다. 그래도 결정이 필요할 때만 사용자에게 묻는다.

## 실행 모드

메인 에이전트 브랜치까지는 어느 모드든 커밋·푸시가 승인 없이 진행된다.

### 모드 A — 대화형 (Agent 도구 + SendMessage) · 기본
- 서브·리뷰 에이전트를 `Agent` 도구로 기동. 피드백 루프는 `SendMessage`로 같은 에이전트를 이어가 컨텍스트를 유지한다. 병렬 구현은 여러 `Agent` 호출을 한 메시지에 담아 동시에 띄운다.

### 모드 B — 자율 (Workflow)
- P4~P6을 `Workflow` 스크립트로 자동·병렬 실행한다. `mode=B` 선택 자체가 Workflow opt-in이다.

## 오케스트레이션 메타 (K와 무관하게 항상 생성)

```
$ORCHESTRATION_META/{루트키}/
├── orchestration.md     # 관제탑 보드(에이전트 상태표·범위 배분·이벤트 로그)
├── agents/{슬러그}.md · review-agent.md   # 계약(불변)
├── reviews/round-{n}/{슬러그}.md
└── agent-logs.json      # {슬러그}→서브에이전트 대화 로그 경로(메인이 수집)
```

- 에이전트 상태값: `대기 → 분석중 → 구현중 → 리뷰중 → 수정중 → 재통합대기 → 완료`(리뷰 에이전트: `대기 → 진행중 → 완료`).
- Phase 전이·리뷰 결과·충돌 판단을 `orchestration.md`의 이벤트 로그에 타임스탬프 1줄로 누적한다.
- **이 메타는 에이전트 수와 무관하게 항상 만든다.** 에이전트가 하나뿐이면 `orchestration.md` 보드는 1행, `agents/`에는 계약 1개(화이트리스트=전체 범위), `reviews/`에는 그 1개 에이전트의 라운드가 쌓인다. K로 생성/생략을 분기하지 않는다.
- 에이전트가 하나뿐이면 루트 키가 곧 그 에이전트의 키이므로, 이 메타와 work 파일(analysis.md 등)이 같은 `$ORCHESTRATION_META/{루트키}/` 폴더에 함께 놓인다(에이전트가 여럿이면 work 파일은 각 하위 에이전트 키 폴더에).

## 승인 게이트

- **승인 불필요(자동)**: 하위/에이전트 브랜치와 상위(루트) 브랜치까지의 커밋·푸시·머지·origin 푸시.
- **⛔ 금지**: `$ORCHESTRATION_DEFAULT_BASE`로의 PR 생성·머지.
- **승인 필요**: 메인 브랜치 밖으로 나가는 반영(배포 등).

## 주의

- **사실 기반 원칙(최우선)**: 모든 판단·분석·배분은 확인된 사실(코드·문서·조회 결과)에만 근거한다. **추측하지 않는다.** 불확실하면 백그라운드 에이전트로 확인한다. 서브·리뷰 에이전트에게도 이 원칙을 전달한다.
- 어려운 용어를 지양하고 쉬운 말로 설명한다.
- 계약(불변)/`status.md`(진행)/`orchestration.md`(관제탑)의 역할을 섞지 않는다.
- 워크트리에는 `node_modules`가 없어 커밋 시 `--no-verify`로 훅을 건너뛴다.
