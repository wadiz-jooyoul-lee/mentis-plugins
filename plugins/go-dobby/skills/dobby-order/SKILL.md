# dobby-order

모든 작업의 **유일한 진입점**이자 오케스트레이터다. 이 스킬을 실행하는 주체가 **메인 에이전트**다. 진입한 작업을 분석해 필요한 에이전트 수(**팬아웃 K**)를 정하고, 서브 에이전트로 착수·구현시키고, 리뷰 루프를 거쳐 상위 브랜치로 통합한다.

> 단독 이슈 해결을 위한 별도 경로는 없다. **단일 버그는 그냥 "에이전트 1명(K=1)"인 오케스트레이션**이다 — 특수 경우가 아니라 팬아웃 수가 1일 뿐이며, 메타·계약·리뷰 등 절차와 산출물은 K와 무관하게 **동일**하다(에이전트·계약·보드 행이 1개가 될 뿐).

## 공통 규칙 (전 Phase 강제 — 위반 = 규약 위반)

아래는 모든 Phase에 적용되는 규칙이다. 각 Phase는 이 규칙을 `(→C#)`로 참조하며, 재기술하지 않는다.

- **C1 커밋·푸시 정책**: 리뷰(P5) 전에는 커밋·푸시하지 않고 워크트리 미커밋 상태로 리뷰한다. blocking=0 통과 후(P6 종료) **처음이자 1회** `commit --no-verify` + 자기 브랜치 푸시(→ 헬퍼 `dobby_commit_push`). 이 커밋·푸시는 **사용자 동의 없이 자동**(전역 "커밋·푸시 동의" 지침의 예외 — 멈춰서 묻지 말 것). ⛔ 정식 배포 베이스(`$ORCHESTRATION_DEFAULT_BASE`, 기본 master)로의 PR·머지는 **금지**(사용자가 직접).
- **C2 K 불변**: K=1도 K≥2와 절차·산출물이 동일하다. 브랜치·계약·리뷰·메타를 K로 분기하지 않는다(에이전트 1명이면 행·계약이 1개가 될 뿐, 루트=에이전트 브랜치라 통합 머지가 자연히 없음 — 특수 분기 아님).
- **C3 상태 단일 정본**: 에이전트 상태는 `orchestration.md`의 `## 에이전트 상태표`가 유일한 정본이다. `status.md`에는 에이전트 상태표를 두지 않는다.
- **C4 상태 선(先)갱신**: 어떤 작업(스폰·전이·리뷰·재통합·완료)이든 **그 전에** 상태표를 먼저 갱신한다. 비활성→활성 진입 시 `착수` 열에 `YYYY-MM-DD HH:MM`(시:분 필수)을 기록한다. 병렬도 에이전트별 개별 갱신(전원 종료까지 미루지 않음). → 헬퍼 `dobby_agent_state`.
- **C5 리뷰어≠구현자**: 코드리뷰는 항상 별도 리뷰 에이전트가 한다(K=1도, 구현자는 셀프리뷰 안 함). 리뷰어 롤은 겸하지 않는다(항상 별도 `리뷰어` 행). 재라운드는 **같은 `review-agent` 슬러그** 재사용 + 하위 키(round-1/round-2). 라운드별 새 슬러그 금지(유령 에이전트 방지).
- **C6 하드 게이트(산출물로 강제)**: ①docs 게이트 `docs-refs.md` ②활성 경로 확정 ③`side-effects.md` — 각 산출물/증명 전에는 다음 단계로 넘어가지 않는다(전이표 ✅).
- **C7 사실 기반**: 모든 판단은 확인된 사실(코드·문서)에만 근거한다. 추측 금지, 불확실하면 백그라운드 에이전트로 확인. 스폰하는 분석/구현/조사 에이전트 프롬프트는 **[공통(전 에이전트 동일: `analysis-discipline.md` 원칙 0·1·2·3, 이중경로면 1·4) → 전용(`role-personas.md`의 그 역할 블록만 → 계약 → 핸드오프 요약)]** 순으로 구성한다(공통 부분은 에이전트 간 동일하게 유지, 역할 블록은 해당 역할만 주입).
- **C8 결정론 갱신은 헬퍼로**: 상태표·이벤트·제목·브랜치·append 등 기계적 갱신은 `dobby-lib.sh` 헬퍼로 한다(전체 통독 없이 부분수정/append). 값(이름·메시지 등)만 인자로 넘긴다.
- **C9 에이전트 배정 = 병렬성 기준 (토큰·속도 절감, 품질 불변)**: 에이전트 경계는 "병렬 가능성"으로 정한다.
  - **순서대로 이어지는 한 흐름(분석→구현)은 한 에이전트**가 이어 맡는다 — **모드 A에서 분석 세션을 종료하지 말고**, P3에서 확정한 계약(화이트리스트)을 `SendMessage`로 전달해 **같은 세션이 그대로 구현으로 이어간다**(구현 역할 블록을 재선언; 코드가 이미 로드돼 있어 재읽기 없음). ⛔ 이는 "메인 인라인"이 아니라 **분석 서브 에이전트가 구현까지 잇는 것**이다(`light` 전용 인라인과 구분). 리뷰는 그래도 **별도**(→C5).
  - **동시에 가능한 독립 영역은 에이전트를 늘려 병렬**(시간 절약).
  - **리뷰는 별도 에이전트 1명이 "먼저 끝난 것부터" 대기열**로 처리한다(영역마다 리뷰→findings 기록→그 diff는 잊고 다음). 영역 **3개↑ + 속도 우선**(급한 오더/모드 B)일 때만 영역별 병렬 리뷰어로 스위치.
  - **⛔ Fallback(=현재 동작, 하한선)**: 분석 세션이 죽었거나 순차 오너십으로 오래 대기해야 하면 **같은 슬러그로 재기동**한다. 이때 전체 문서 대신 **짧은 핸드오프 요약**(건드릴 파일·핵심 결정·범위)을 프롬프트에 실어 재읽기를 줄이되, 불확실하면 원문을 확인한다(정보 손실 금지).
  - **모드 B(Workflow)**는 세션 이어가기가 없으므로 파이프라인(분석→구현을 아이템별로 겹쳐 실행)으로 대체한다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/go-dobby/config.env`를 source 해 환경 변수를 **읽기만** 한다. ⛔ **이 스킬은 config.env를 저장·수정·생성하지 않는다**(값 변경은 `dobby-init` 전용 — config.md '비파괴 원칙'). config.env 파일이 아예 없으면 멈추고 `/dobby-init`을 먼저 실행하도록 안내한다. 메타 루트 `ORCHESTRATION_META`, 변수 목록·기본값, 폴더 배치(워크트리 `$ORCHESTRATION_WORKSPACE/subtree/` · 메타 `$ORCHESTRATION_META/`)는 모두 그 문서에 있다. 이하 메타 경로는 `$ORCHESTRATION_META` 기준.

## 스킬 관계 (building block)

```
dobby-init    ← 환경 설정 전용(config.env를 만드는 유일한 스킬). 최초 1회·설정 변경 시.
dobby-order   ← 오케스트레이터(메인). 전체 흐름 지휘.
  ├─ dobby-start    ← 에이전트별 착수·분석. 메인이 P1에서 실행.
  ├─ dobby-impl     ← 에이전트별 구현(work-type=code). 서브 에이전트가 P4에서 실행.
  └─ dobby-produce  ← 에이전트별 산출(work-type=비소스: 문서·리서치 등). P4 대안.
```

> **설정 전제:** dobby-order를 비롯한 모든 작업 스킬은 `config.env`를 **읽기만** 한다. **config.env가 없으면**(최초 실행) 진행하지 말고 **`/dobby-init`을 먼저 실행**하도록 안내한다. 설정값 생성·변경은 오직 `dobby-init`이 한다(config.md '비파괴 원칙').

리뷰·검증·정리:
- **code 작업**: `dobby-test`(브라우저 검증) → `dobby-resolve`(해결 표시) → `dobby-end`(정리).
- **비소스 작업**: 브라우저 테스트 없음 — P5 내용 리뷰로 검증하고 → `dobby-resolve` → `dobby-end`(meta 산출이면 워크트리 정리 no-op).
- 이들 중 dobby-test·resolve·end는 통합 이후 사용자가 실행한다.
- **QA 감시**: 통합(P7) 이후 `dobby-qa {루트키}`로 진입하면, Jira를 폴링해 이 오케스트레이션 소관 버그를 잡아 **P8(후속 재개) 흐름으로 자동 처리**한다(담당 에이전트 수정 → 담당 리뷰어 리뷰). 로컬 세션 루프로 동작.

## 입력 / 진입 모드

`/dobby-order {대상} [base=] [subtasks=] [agents=] [mode=] [light] [produce] [ref=키]`

> **`/dobby-order help`(또는 `--help`·인자 없음)**: 오케스트레이션을 시작하지 말고, 아래 **"## 도움말(help) 출력"** 의 블록을 그대로 출력하고 종료한다(다른 작업·조회 없이).

- **(a) 이슈 키/URL**: 예 `FE1-1187` 또는 `$JIRA_BASE_URL/browse/FE1-1187`. 이슈 안의 기획·피그마·디자인을 확인해 진행.
- **(b) 이슈 + 기획서/피그마**: 이슈와 함께 문서·피그마 링크를 받아 진행.
- **(c) 문서 전용(이슈 없음)**: 특정 문서만 주고 작업을 요청. 이슈 키가 없으므로 **작업 키를 생성**한다.
  - 규칙: **`TASK-{slug}`** — `slug`는 문서 제목/주제에서 뽑은 짧은 영문 kebab. 충돌 시 `-YYYYMMDD` 부가(예: `TASK-login-refactor`).
  - 이 키가 이슈 키 자리를 그대로 대체한다: 폴더 `$ORCHESTRATION_META/TASK-{slug}/`, 브랜치 `feature/TASK-{slug}`.
  - **Jira 조회·상태 전환은 건너뛴다.** 생성한 작업 키는 시작 계획 게이트(아래 "확인 게이트" #1)에서 사용자에게 1회 확인한다.

인자:
- `base=`: 상위 브랜치의 베이스(미지정 시 `$ORCHESTRATION_DEFAULT_BASE`, 기본값 master).
- `subtasks=키,키`: 대상 하위이슈 명시(일부만). `agents=N`: 팬아웃 강제. 미지정 시 **자동 판단**.
- `light`: **경량(인라인) 모드**. 소규모 단일 이슈를 서브 스폰 없이 메인이 착수·분석·구현까지 인라인 처리하고 **리뷰만 별도 에이전트**로 돌린다(아래 "P4-L 경량 분기"). **명시할 때만 발동**하며 자동 전환되지 않는다. **`light`는 구현 단계를 인라인으로 강제하므로 그 단계는 `mode=B`(Workflow)의 대상에서 제외된다** — light에서 `mode=B`를 줘도 Workflow는 리뷰 스폰에만 적용되고 구현은 메인 인라인으로 수행한다.
- `mode=A|B`: 실행 모드(미지정 시 A).
- `produce`: **비개발(조사·산출) 강제 모드.** 이슈가 코드 작업이라도 **코드는 짜지 않고**(소스는 읽기 전용) 비소스 산출물만 만든다 — 전 에이전트 `work-type=비소스` 고정, P4는 `dobby-produce`만, 검증은 P5 내용 리뷰(브라우저 X). **뒤에 자유 문장으로 목적을 주면**(예: `produce 기획서 보강`) 그 산출을, **아무 말 없으면 기본 = "구현 가능 여부 판단(타당성 조사)"** 리포트를 만든다(아래 "produce 모드" 참조).
- `ref=키`: **참조할 조사 이슈 키.** 다른 이슈에서 `produce`로 한 조사(개발 인계)를 **재활용**해 재조사를 줄이고 빠르게 개발한다(아래 "ref= 조사 재활용").

## 도움말(help) 출력
`help`/`--help`/인자 없음이면 아래를 **그대로 출력**하고 종료한다(구구절절 설명 금지 — 옵션당 한 줄 + 예시):

```
dobby-order — 이슈/작업을 분석해 개발·리뷰·통합까지 자동 진행

기본
  /dobby-order FE1-1274            이슈 하나 진행(분석→구현→리뷰→통합)
  /dobby-order <URL|문서경로>       이슈 URL·문서로도 시작(문서면 TASK- 키 생성)

자주 쓰는 옵션 (필요할 때만)
  produce      개발 대신 조사만. 그냥 붙이면 "구현 가능 여부" 리포트가 기본.
               예) /dobby-order FE1-1274 produce
                   /dobby-order FE1-1274 produce 기획서 보강해줘   (뒤에 원하는 걸 말로)
  ref=키       다른 이슈에서 한 조사를 참조해 재조사 없이 빠르게 개발.
               예) /dobby-order FE1-1300 ref=FE1-1274
  light        작고 단순한 이슈를 가볍게(메인이 직접 구현, 리뷰만 분리).
               예) /dobby-order FE1-1274 light
  mode=A|B     A=대화형(기본), B=자율(Workflow로 자동·병렬).
               예) /dobby-order EPIC-1 mode=B

기타 옵션
  agents=N        에이전트 수 강제(기본 자동 판단)
  subtasks=키,키  대상 하위이슈 일부 지정
  base=브랜치     상위 베이스 브랜치(기본 master)

한눈에
  될지만 판단 → produce      조사→개발 → ref=조사키
  빠른 소규모 → light        자동 병렬 → mode=B
```

## produce 모드 (비개발 강제 — 조사·산출)
- **코드 미변경**: 소스는 근거 확인용으로 **읽기만** 하고 수정·커밋·브랜치·머지를 하지 않는다. 전 에이전트 `work-type=비소스`로 고정(P3의 자동 분류를 건너뛴다), P4는 `dobby-produce`, 검증은 **P5 내용 리뷰**.
- **목적 결정**: `produce` 뒤 자유 문장이 있으면 그 산출(기획 보강·자료 조사 등), 없으면 **기본 = 구현 가능 여부 판단(타당성 조사)**.
- **기본 산출물(타당성)**: `deliverables/feasibility.md` — 형식은 `dobby-produce`의 "타당성/개발 인계 형식"을 따른다(결론·근거·필요 변경 범위·리스크·**개발 인계**·권고). 개발 인계 섹션은 다음 개발이 재조사를 덜 하도록 활성 경로·변경 범위 후보·구현 방향 초안·리스크를 담는다.

## ref= 조사 재활용 (조사 → 개발 인계)
- 개발 오더에 `ref={조사키}`를 주면, `$ORCHESTRATION_META/{조사키}/`의 **조사 산출물(feasibility.md·produce.md·analysis.md의 "개발 인계")을 먼저 읽어 분석 씨앗으로 재활용**한다: 활성 경로·변경 범위·리스크를 처음부터 다시 파지 않고 **사실만 빠르게 재확인**한 뒤 그 범위로 P3/P4에 진입한다(재조사 최소화).
- `ref=` 없이 **같은 키를 이어서** 개발하면 자기 폴더의 이전 조사를 동일하게 재활용한다.
- 분석 산출물(analysis.md)에 **출처를 남긴다**(예: `근거: {조사키} 조사`). 참조한 조사가 없거나 형식이 안 맞으면 **경고만 남기고 정상 분석으로 진행**(그레이스풀).

## 진입 즉시 (골격 status.md 먼저 기록)

**팬아웃 판단·Explore·워크트리 생성보다 먼저**, 대상에서 루트 키를 정하는 즉시 최소 골격 `status.md`를 남긴다. 대시보드 목록 등장 조건은 `$ORCHESTRATION_META/{루트키}/`에 `status.md`(또는 `orchestration.md`)가 있는 것뿐이므로, 이 한 파일만 있으면 **바로** 목록에 뜬다. 이게 없으면 팬아웃 분석·워크트리·origin 푸시가 다 끝나는 P0까지(수 분) 목록에 안 떠서 "시작이 안 된 것처럼" 보인다.

- 루트 키: 이슈 키/URL이면 그 키. 문서 전용이면 위 `TASK-{slug}` 규칙으로 제목만 보고 **즉시** 생성한다(Jira 조회 불필요).
- `mkdir -p $ORCHESTRATION_META/{루트키}` 후 `status.md`에 **아는 것만** 최소로 기록한다(조회 전이므로 제목은 대상 원문이라도 무방). **→ 헬퍼 `dobby_scaffold_meta {루트키} "{제목}"`**(폴더+골격 status.md):

  ```markdown
  # {루트키}

  ## 이슈/작업
  - **제목**: {이슈 요약 / 문서 제목 / 대상 원문 중 아는 것}

  ## 현재 단계
  - **단계**: 착수
  - **갱신**: {YYYY-MM-DD HH:MM}
  ```

- 이후 P0(`orchestration.md`)·P1(`dobby-start`의 status.md 초기화)에서 이 파일을 정식 스키마로 채워 나간다. 초기값 `착수`는 대시보드가 진행중 단계로 인식하므로 별도 처리는 필요 없다.
- ⛔ **이 골격 제목은 임시값**이다(이슈 키/slug 원문). **이슈를 조회한 직후(아래 P1) 반드시 `dobby_set_title`로 실제 요약으로 갱신**한다 — dobby-start를 거치지 않는 light·K=1·TASK 흐름에서도 예외 없다.

## 팬아웃 K 결정 (진입 판단)

이슈 타입이 아니라 **코드 범위(파일 오너십)**로 판단한다. 타입·하위이슈는 강한 힌트일 뿐이다.

> **⛔ docs 게이트 (P1 최우선·차단 — 강제, →C6):** 조회·Explore·팬아웃 판단을 시작하기 **전에**, 가장 먼저 **`dobby_docs_gate {루트키} "kw1|kw2"`**(config.md '공용 헬퍼')를 실행한다. 이 헬퍼가 `$ORCHESTRATION_META/{루트키}/docs-refs.md`를 생성하고 관련 문서 경로를 반환한다.
> - **히트가 있으면 → 그 문서를 반드시 먼저 읽어** 기본 위치·배경·플로우·API 계약을 파악한 뒤에만 Explore/분석/팬아웃으로 진행한다. 읽은 문서 경로·요지를 이후 **모든 Explore/분석 에이전트 프롬프트에 실어** 보낸다(analysis-discipline 원칙 0).
> - 히트가 없거나 DOCS_ROOT가 없어도 `docs-refs.md`에 "확인함"으로 기록돼 게이트를 통과한다(조용한 스킵 방지).
> - **⛔ `docs-refs.md`가 생성되기 전에는 Explore·분석·팬아웃·P4로 넘어가지 않는다.** dobby-start를 거치지 않고 pre-Explore를 직접 띄우는 경로에서도 예외 없다.

1. 이슈/문서 조회 + 하위이슈 탐색(`parent = {상위}` JQL) + (멀티 repo면) repo별 **읽기 전용 Explore 에이전트**로 범위를 사실 기준으로 분석한다.
   - **⛔ status.md 제목 갱신(필수 · 조회 직후 · dobby-start 무관)**: 이슈 요약(문서 전용이면 확정한 작업 제목)을 받은 **바로 이 자리에서** 골격 status.md의 임시 제목(키/slug)을 **실제 요약으로 갱신**한다. **→ 헬퍼 `dobby_set_title {루트키} "{실제 제목}"`**. light·K=1·TASK 등 dobby-start를 거치지 않는 흐름에서도 반드시 한다.
   - **이슈 원문 저장(재활용 · 대시보드 Jira 탭용)**: 루트 키가 실제 Jira 이슈면(문서 전용 `TASK-` 제외), **방금 조회한 이슈 원문(설명 본문)을 그대로** `$ORCHESTRATION_META/{루트키}/jira-issue.md`에 저장한다. 이미 읽은 내용을 파일로 쓰는 것뿐이라 **추가 조회·분석·요약은 하지 않는다**(메인 진행에 부담 없음). 설명이 비어 있으면 저장하지 않는다. 정리·코멘트 요약·업데이트 보강·게시는 **대시보드 버튼**이 별도 스킬(`dobby-jira-tab`)로 처리하므로, 여기서는 **원문 저장만** 한다.
   - **⛔ Jira 상태 → 진행 중 전환(필수 · dobby-order가 직접)**: 루트 키가 실제 Jira 이슈면(문서 전용 `TASK-` 제외), 이슈를 조회한 **바로 이 자리에서** 상태를 진행 중으로 전환한다. **dobby-start에 미루지 않는다**(단일/`light`/인라인 흐름에선 dobby-start가 안 뜨거나 메인이 이미 조회해 전환이 누락될 수 있다).
     - **`statusCategory.key`로 판정**: `new`(할 일)면 `mcp__mcp-atlassian__jira_get_transitions`로 도착 상태 `statusCategory`가 `indeterminate`인 전이를 골라 `mcp__mcp-atlassian__jira_transition_issue`로 전환한다. `indeterminate`(이미 진행 중)면 그대로 둔다. `done`(완료)이면 전환하지 말고 사용자에게 확인한다.
     - 하위이슈로 팬아웃하는 경우, 각 대상 하위이슈도 착수 시 동일하게 진행 중으로 전환한다(dobby-start가 그 대상을 맡으면 dobby-start의 1단계가, 메인이 직접 조회했으면 여기서).
     - 도착 상태 이름은 프로젝트마다 다르므로 **이름이 아니라 `statusCategory`로 판정**한다(하드코딩 금지). 애매하면 사용자에게 확인.
2. 분리 가능한 오너십 영역을 도출한다:
   - 병렬 가능한 영역이 **2개 이상** → `K = 영역 수`(다중). 영역과 하위이슈를 **1:1**로 맞춘다(없으면 그 분해로 하위이슈 생성).
   - **한 덩어리(cohesive)** → `K=1`(단일). 하위이슈 불필요.
   - **멀티 repo → repo당 최소 1 에이전트**(repo 단위 오너십 = 통합 시 머지 충돌 원천 차단).
3. 판단에 필요한 사실이 불확실하면 **묻지 말고 백그라운드 에이전트로 직접 확인**한다(추측 금지).

### 브랜치 토폴로지 (하나의 규칙 — K로 분기하지 않는다)

- **루트 브랜치** `{prefix}/{루트키}`를 `{base}`에서 만든다. 통합 지점이다.
- **에이전트 브랜치**: 에이전트키 ≠ 루트키인 에이전트는 `{prefix}/{에이전트키}`를 루트 브랜치에서 만들어 작업하고, 리뷰 통과 후 **루트 브랜치로 머지**한다.
- 에이전트가 하나뿐이면 그 에이전트키 = 루트키이므로 **루트 브랜치가 곧 에이전트 브랜치**다 — 추가 브랜치도, 통합 머지도 자연히 없다(별도 부모 브랜치를 만들지 않는다). 이는 "에이전트 수 1"의 자연스러운 귀결일 뿐 특수 분기가 아니다.
- `{base} └ {prefix}/{루트키} [├ {prefix}/{에이전트A} └ {prefix}/{에이전트B} …]` (대괄호 부분은 에이전트가 여럿일 때 생긴다.)

## 절차 (Phase)

전이는 **객관적 조건**으로 자율 진행한다(매 단계 확인하지 않는다). 아래 "전이 조건표"·"확인 게이트" 참조.

### P0. 셋업
- 소스 루트에서 `git fetch origin {base}`.
- **루트 브랜치·워크트리 생성** + origin 푸시. **→ 헬퍼 `dobby_setup_worktree {repo} {루트키} {prefix} {base}`**(base 로컬/원격 자동 감지·재사용·경로 반환) + **`dobby_ensure_board {루트키}`**(상태표·이벤트로그 골격). (→C2)
- **메타 루트는 항상 동일**: 루트 키 폴더 `$ORCHESTRATION_META/{루트키}/`에 `orchestration.md`·`agents/`·`reviews/`를 **항상** 둔다(→C2, 규격은 아래 "오케스트레이션 메타"). 하위 에이전트 키가 따로 있으면 그 키 폴더(`$ORCHESTRATION_META/{에이전트키}/`)에는 work 파일이 들어간다.
- **멀티 repo면 저장소마다 위를 반복.** base는 저장소마다 다를 수 있다(예: FE master, BE develop).

### P1. 착수·분석
- **⛔ 활성 경로 확정 게이트(P1 최우선·차단, →C6)**: 분석을 시작하기 전에, 대상 기능/URL이 **실제로 어느 앱·엔트리·라우트에서 렌더·실행되는지**를 라우트 등록·엔트리 빌드·마운트 지점으로 **증명**한다. 국내가 `static/entries/*`(레거시)와 `apps/global`(이관본) **이중화**돼 동일 기능이 양쪽에 중복 존재하므로, 실행되는 쪽만 근거로 삼고 안 되는 쪽은 **데드코드로 명시·제외**한다. URL 패턴 매칭이 관건이면 실제 URL을 정규식에 대입해 매칭을 확인한다. **확정하지 못하면 거기서 멈추고 사용자에게 보고**한다(추측으로 한쪽 선택 금지). **이 게이트를 통과하기 전에는 P4 구현으로 넘어가지 않는다.**
- **⛔ 국내/글로벌 중복·충돌 처리 원칙(이중 경로)**: 동일 기능이 국내(`static` 레거시)와 글로벌(`apps/global`) 양쪽에 존재해 **중복·충돌**이 발견되면, ①활성 경로 확정 → ②중복의 근원 규명(git 히스토리로 이관 시 가드 누락 등 파악) → ③**두 구현의 정보·동작이 실제로 동일한지 실측으로 검증**(단일·멀티·엣지 케이스, 실제 렌더 결과 기준)한다. **동일하면** 국내 레거시를 제거하고 신규 글로벌(공통)으로 통일한다. **다르면 통일하지 말고 `LocaleSettings`(`@wadiz/settings`)로 국내/글로벌 분기를 추가·유지**한다 — 함부로 한쪽으로 합치면 회귀가 난다. **통일은 기본값이 아니라 동일성 검증을 통과했을 때만의 선택**이다. 레거시 제거로 사라지는 UX(더보기/접기 등)는 신규 공통 쪽으로 이식해 보존하고, i18n 하드코딩 문구는 공통 키를 우선 재사용한다.
- 스폰하는 모든 분석/구현/조사(Explore) 에이전트 프롬프트에 `analysis-discipline` 원칙 + `role-personas` 역할 블록을 주입한다(→C7). 분석 에이전트는 "착수·분석 에이전트" 블록. 전제는 사실이 아니라 검증 대상임을 명시해 확증 편향을 차단한다.
- **⛔ 분석 에이전트도 스폰 즉시 보드에 등록(→C4)**: `dobby-start`를 띄우기 **직전**, 그 슬러그 행을 상태표에 **상태 `분석` + 착수 시각**으로 추가하고, 스폰 **직후** `agent-logs.json`에 로그 경로를 기록한다. 분석이 끝나면 같은 슬러그 행이 그대로 `구현`으로 이어진다(P3에서 계약·화이트리스트만 보강). (슬러그가 아직 안 정해졌으면, 팬아웃 판단으로 정한 대상 단위로 잠정 슬러그를 부여해 등록한다.) **→ 헬퍼: 행 추가 `dobby_agent_add {루트키} {슬러그} {이름} "{설명}" 분석`, 로그 `dobby_log {루트키} {슬러그} {output_file}`, 이벤트 `dobby_event {루트키} "P1 착수 — ..."`, 전이 `dobby_agent_state {루트키} {슬러그} 구현`.**
- 각 대상(하위이슈 또는 단일 이슈/작업)에 `dobby-start {키} base={상위 브랜치 또는 base}` 실행 → 워크트리 생성·코드 분석·`analysis.md`·`test-plan.md` 확보.
- **하위이슈가 없거나 새로 만들 때**: 먼저 P3 범위 배분으로 **에이전트 단위 분해**한 뒤 1:1 하위이슈를 생성하고 각각 dobby-start를 실행한다(이 경우 P3가 P1보다 먼저).
- 착수 전 각 저장소 원격에서 상위이슈 관련 프로토타입/스파이크 브랜치를 검색해, 재사용 가능분만 **사실 기준**으로 계약에 반영한다.

### P2. 분석 검증
- 분석 산출물을 숙지하고 **기획서·Jira·(있으면) Figma/문서**와 교차 검증한다. 어긋나면 해당 dobby-start 분석을 보정한다.
- 미결 결정이 있으면 목록화한다. **코드로 확인 가능한 건 확인해 정하고**(백그라운드 에이전트), 순수 제품/디자인 결정만 사용자에게 승인받는다. 미결을 남긴 채 P3로 넘어가지 않는다.

### P3. 범위 배분 + work-type 분류
- **구현에 필요한 코드를 먼저 분석**해 에이전트별 **수정 파일 화이트리스트**를 정한다.
- 오너십이 안 겹치면 → **병렬**로 바로 P4(사용자에게 묻지 않음). 겹치면 → **순차**로 자동 결정(사유 기록).
- 멀티 repo면 각 에이전트는 **한 저장소에만** 속하게 배분하고, 저장소 간 결합점은 **교차 저장소 API 계약** 하나로 모아 먼저 확정한다.
- **work-type 분류(라우팅)**: 각 에이전트 작업을 분류한다.
  - `work-type = code` → 구현은 `dobby-impl`, 검증은 `dobby-test`.
  - `work-type = 비소스`(문서·리서치·분석·설정 등) → 산출은 `dobby-produce`, 검증은 **P5 내용 리뷰**(브라우저 테스트 없음). 산출물이 repo면 자기 브랜치 푸시, repo 밖이면 `$ORCHESTRATION_META/{키}/deliverables/`에 저장.
  - 한 오케스트레이션에 두 work-type이 섞일 수 있다(에이전트별로 다르게 라우팅).
- **`orchestration.md`·`agents/{슬러그}.md`(에이전트별 계약)·`agents/review-agent.md`를 항상 작성한다**(→C2). 계약에 수정 허용 범위(화이트리스트)·공유 데이터 계약을 명시한다. **에이전트가 하나뿐이면 계약도 1개**이며, 그 화이트리스트는 **전체 범위**(오너십 제약 없음)로 적는다. 보드(orchestration.md)는 에이전트 1행이 된다.
- **⛔ 사이드 이펙트 분석(`side-effects.md`) — P4 진입 게이트(→C6)**: 구현 설계가 굳으면(화이트리스트·계약 확정), 그 설계가 유발할 수 있는 파급을 **다각도로 점검**해 `$ORCHESTRATION_META/{루트키}/side-effects.md`에 정리한다. 오케스트레이터가 각 에이전트 분석(analysis.md)을 근거로 **1개로 consolidate**한다(형식·각도는 아래 "사이드 이펙트 분석" 규칙).

### P4. 구현
- **선행 조건: P1 활성 경로 확정 게이트 + P3 사이드 이펙트 분석(`side-effects.md`)을 통과·작성했어야 한다.** 미완이면 P4로 진입하지 않는다.
- **구현 에이전트 기동(→C9)**: 순서상 분석→구현은 한 에이전트다 — 모드 A에서 **P1 분석 서브 세션이 살아 있으면 그 세션을 이어서 구현**시킨다(`SendMessage`로 확정 계약 전달 + 구현 역할 블록 재선언). 세션이 없거나(종료·재기동) 순차 대기가 길면 같은 슬러그로 **재기동**(핸드오프 요약 전달). 에이전트가 하나뿐이어도 **메인이 직접 구현하지 않는다**(리뷰 분리·경로 단일화 — `light`만 예외로 P4-L). 재기동 시 **계약 파일 경로(`agents/{슬러그}.md`)를 첫 지시로 전달**하고, 지시에 `analysis-discipline` 원칙 + `role-personas` 역할 블록(code는 "구현 에이전트", 비소스는 "비소스 산출 에이전트")을 주입한다(→C7). 독립 영역이 여럿이면 에이전트를 늘려 병렬(→C9).
- work-type으로 실행 스킬을 정한다:
  - `code` → `dobby-impl`로 구현 → `status.md` `구현`·`implementation.md` 기록. **⛔ 이 시점엔 커밋·푸시하지 않는다** — 변경을 **워크트리에 그대로 둔 채** 메인에 리뷰를 요청한다.
  - `비소스` → `dobby-produce`로 산출 → `산출`·`produce.md` 기록. repo 산출도 **커밋·푸시하지 않고** 워크트리 상태로 리뷰 요청(meta 산출이면 `deliverables/`에 저장 — 커밋 없음).
- 어느 쪽이든 끝나면 메인에 리뷰 요청·대기. (커밋·푸시 시점은 →C1: 코드리뷰는 미커밋 변경 대상, 커밋은 blocking=0 통과 후 1회.)

### P4-L. 경량(인라인) 분기 — `light` 플래그일 때만
`light`가 지정된 소규모 단일(K=1) 이슈에 한해, **P4에서 서브 에이전트를 스폰하지 않고 메인이 착수·분석·구현을 인라인으로 수행**한다. **위 P4의 "반드시 서브로 기동"에 대한 명시적 예외**이며, **리뷰어≠구현자는 P5 리뷰 에이전트로 그대로 지킨다(→C5).** 자동 발동하지 않는다.

- **적용 조건(발동 게이트)**: `light` 플래그 + K=1 + 소규모(수정 예상 파일 소수·단일 영역·단일 repo). 분석 결과 여러 영역/저장소로 번지면 **경량을 중단하고 일반(서브) 모드를 권한다**(메인 컨텍스트 포화 방지).
- **인라인 실행**: 메인이 dobby-start/dobby-impl을 **서브로 띄우지 않고 그 절차를 직접 수행**한다 — **착수 시 docs 빠른 키워드 검색**(dobby-start 5단계: 이슈 키워드로 `$ORCHESTRATION_DOCS_ROOT` grep → 관련 문서만 열기) → 활성 경로 확정 → `analysis.md`·`test-plan.md` → 계약 허용 범위(단일이라 전체) 안에서 구현 → `implementation.md`. 커밋·푸시는 여기서 하지 않는다(→C1). 프롬프트 규율로 `analysis-discipline` 원칙 + `role-personas`의 「경량 인라인 에이전트」 블록을 따른다(→C7).
- **리뷰(P5) — 미커밋 기준**: 구현 후(커밋 전) 별도 리뷰 에이전트를 스폰하되 **이 인라인 워크트리 경로를 전달**(`--add-dir`)해 **미커밋 변경(`git diff {fork}`)** 을 리뷰한다(「리뷰 에이전트」 블록). 피드백은 메인이 워크트리에서 인라인 반영(P6) → blocking=0까지 재리뷰. 이후 커밋·푸시(→C1).
- **⛔ 대시보드 호환 필수 산출물(빠지면 상세 페이지가 조용히 빈다)**:
  - `status.md`: `## 현재 단계`, `## 이슈/작업` 메타, `## 세션`(세션 ID 36자 UUID·작업 경로 cwd). 에이전트 상태표는 두지 않는다(→C3).
  - `orchestration.md`: **`## 에이전트 상태표`(→C3)에 인라인 에이전트 1행(+리뷰 시 리뷰 에이전트 행)** — `슬러그`·`이름`·`설명`·`상태`·`라운드`·`착수`·`갱신` 컬럼(인라인 에이전트 이름은 **`분석/개발자`**, 리뷰는 **`리뷰어`**) · `## 이벤트 로그`(기존 K=1과 동일하게 메인이 작성).
  - `agent-logs.json`: `{ "{인라인슬러그}": "~/.claude/projects/{인코딩cwd}/{세션}.jsonl", "review-agent": "{리뷰 output_file}" }` — **인라인 슬러그는 메인 세션 전사를 가리킨다**(대시보드가 메타 경로는 자동 제외하므로 코드변경·콘솔이 정상 추출된다).
  - `implementation.md`: work-type=code 확정 + 보드 "구현" 섹션 렌더용.
- **상태 전이(5상태 모델)**: 인라인 에이전트 행 상태를 `분석 → 구현 → 완료`로(리뷰 도는 동안에도 인라인 에이전트는 `구현` 유지 — 자기 상태만), 리뷰 에이전트 행을 `리뷰 → 완료`로 갱신한다. 상태 선(先)갱신 원칙을 그대로 지킨다(→C4). 이후 P7 통합·`dobby-explain`은 일반 흐름과 동일.
- **결과**: 목록·보드·콘솔·코드변경·구현내용·검증 렌더는 일반 K=1과 동일. 유일한 차이는 인라인 에이전트의 콘솔·diff가 격리된 서브 로그가 아니라 **메인 세션 전사**에서 나온다는 점뿐이다.

### P5. 코드리뷰 (오케스트레이터 단일 리뷰)
- **별도 리뷰 에이전트**가 각 에이전트 **워크트리의 미커밋 변경**을 **적대적으로** 리뷰한다(K=1도 동일 — →C5, 구현자 dobby-impl은 셀프리뷰 안 하므로 이 리뷰가 유일한 코드리뷰다). **기본은 리뷰어 1명이 "먼저 끝난 구현부터" 대기열로**(영역마다 리뷰→findings 기록→그 diff는 잊고 다음 — →C9), 영역 3개↑+속도 우선일 때만 병렬 리뷰어. 저장소마다 그 repo 규약(계층형 CLAUDE.md·리뷰봇 규칙)으로 리뷰한다. 리뷰 에이전트 프롬프트에 **`role-personas.md`의 "리뷰 에이전트" 블록**을 삽입한다(→C7).
- **⛔ 리뷰 접근(B안)**: 미커밋 변경은 fetch/pull이 안 되므로, 리뷰 에이전트에 **그 에이전트의 워크트리 경로를 전달**(`--add-dir`)해 **같은 워크트리 안에서 직접** diff·파일을 읽게 한다(같은 머신 전제).
- 리뷰 대상 diff: **`git diff {fork}`** (워크트리의 미커밋 변경 포함; `fork` = `git merge-base origin/{base} HEAD`). 아직 커밋 전이라 `..HEAD`를 쓰지 않는다.
- **리뷰 루브릭**(리뷰 에이전트에 전달):
  - **A. 정확성/품질**: 버그·논리 오류, null/undefined·경계값, 비동기/에러 처리, 회귀(소비처 영향), 불필요한 복잡성·데드코드, 설계-구현 불일치.
  - **B. 규약 적합성**: 그 워크트리·영역의 CLAUDE.md/NAMING/GLOSSARY를 **직접 읽어** 대조.
  - **C. CI 고유 규칙**: `.github/workflows`에서 읽어 반영(예: `packages/i18n/**`는 키 rename/삭제만 지적).
  - **D. 커밋 메시지(커밋 시점 검사)**: B안에선 커밋이 리뷰 통과 후에 생기므로 코드리뷰 단계엔 검사할 커밋이 없다. **리뷰 통과 후 커밋할 때** 규칙을 지키고 오케스트레이터가 커밋 직후 확인한다 — `fix:`/`feat: {키} ...` 의미 단위, **오케스트레이션 내부 용어 유출 금지**(`round-N`·"리뷰 반영"·라운드/슬러그/메타 경로 등) + **저장소가 금지하는 서명**(`Co-Authored-By`·"Generated with …") 금지.
  - **E. 범위 준수**: 각 에이전트가 계약 화이트리스트 범위만 수정했는지 검증(오케스트레이터 리뷰만 할 수 있는 확인). 에이전트가 하나뿐이면 화이트리스트가 전체 범위라 이 항목은 사실상 통과지만, 검사 절차 자체는 동일하게 수행한다.
  - 심각도: **blocker**(병합 불가) / **major**(강권) / **minor·nit**(사소).
- **비소스 작업(work-type=비소스)**은 위 코드 루브릭 대신 `dobby-produce`의 **"내용 리뷰 기준"**(정확성·완결성·요청 부합·근거 명시·규약)으로 리뷰한다.
- 결과를 **항상** `$ORCHESTRATION_META/{루트키}/reviews/round-{n}/{슬러그}.md`에 심각도·`파일:라인`·수정 제안으로 남긴다(에이전트 수와 무관). **→ 헬퍼 `dobby_review_path {루트키} {n} {슬러그}`**(폴더 생성·경로 반환; 리뷰 findings 내용은 LLM이 작성).

### P6. 피드백 루프
- 리뷰 결과를 해당 서브 에이전트에 전달(모드 A는 `SendMessage`로 같은 에이전트 이어감). `dobby-impl`이 **워크트리에서 그대로 반영**(아직 커밋 안 함) → 재리뷰(같은 워크트리 미커밋 diff). 라운드 중에는 커밋·푸시하지 않는다(→C1).
- **모든 수정 라운드 후 반드시 재리뷰(생략 금지).** blocking(blocker·major)=0이 될 때까지 반복하되, 상태는 **구현 에이전트=`구현` 유지 · 리뷰 에이전트=`리뷰`**(재리뷰마다 다시 `리뷰`). **최대 라운드 기본 3**, 초과 시 사용자 에스컬레이션.
- **⛔ 리뷰 통과(blocking=0) 후 커밋·푸시(→C1)**: 이때 **처음으로** `commit --no-verify`(리뷰 통과분 전체를 의미 단위 커밋, 위 D 메시지 규칙 준수) → 자기 브랜치 푸시. **→ 헬퍼 `dobby_commit_push {워크트리} {브랜치} "{메시지}"`**(메시지 문구만 LLM이 작성). 그 뒤 **곧바로 P7 통합**으로 넘어가 상태를 `완료`로 갱신한다. (비소스 repo 산출도 동일: 리뷰 통과 후 커밋·푸시. meta 산출은 커밋 없음.)

### P7. 통합
- 리뷰 클린 → 각 에이전트 브랜치를 **루트 브랜치로 순차 머지·푸시**한다(멀티 repo면 저장소별 루트 브랜치로 각각). **→ 헬퍼 `dobby_merge_root {워크트리} {루트브랜치} {에이전트브랜치}`**. 에이전트가 하나뿐이면 그 브랜치가 곧 루트 브랜치라 머지할 것이 없다(→C2).
- **통합 직후 그 구현 에이전트 상태를 `완료`로 갱신한다**(→C3·C4) — 상태표를 `구현 → 완료`로 바꾸고 갱신 일시를 찍는다. 리뷰 에이전트도 리뷰가 끝났으면 `완료`.
- **⛔ 정식 배포 베이스로의 PR 생성·머지는 금지(→C1).** 사용자가 요청해도 이 스킬 범위에선 하지 않고, 별도 절차로 사용자가 직접 진행하도록 안내만 한다.
- **통합 완료 후 `dobby-explain {루트키}`를 실행**해 비전공자용 **구현 내용 문서**(`explainer.md`)를 자동 생성한다. 실제 산출물(analysis·implementation/produce·orchestration·리뷰·에이전트 로그)만 근거로 정리한다(지어내기 금지). 대시보드 "구현 내용" 페이지가 이 파일을 렌더한다.
- **통합 후 사용자 수동 확인 가이드(`test-guide.md`) 작성**: 실제 변경(diff)·각 에이전트 `implementation.md`/`produce.md`·`side-effects.md`를 근거로, **사용자가 직접 사이드이펙트를 확인**할 수 있는 **수동 TC 목록**을 `$ORCHESTRATION_META/{루트키}/test-guide.md`에 정리한다(자동 브라우저 테스트인 dobby-test와 별개). 오케스트레이터가 **1개로 consolidate**한다(형식은 아래 "확인 가이드" 규칙). 새 기능 확인 TC와 **인접 기능 회귀 확인 TC**를 함께 담되 **회귀 확인을 반드시 포함**한다.

### P8. 후속 작업 (완료 후 사용자 피드백 재개)

1차 통합(P7)·`dobby-resolve` 이후 **사용자 피드백으로 후속 작업이 생기면**, 새로 시작하지 않고 **그 범위를 소유한 담당 에이전트가 이어서** 처리한다. 절차·규칙·산출물은 1차와 **동일**하다(축소·생략 없음).

- **담당 라우팅**: 후속 작업이 건드리는 파일·코드 오너십을 보고, **그 범위를 소유한 기존 에이전트(계약)** 에게 맡긴다. 임의로 새 에이전트를 만들거나 메인이 직접 고치지 않는다.
  - 모드 A면 `SendMessage`로 **같은 에이전트를 이어가고**(구현 컨텍스트 유지), 세션이 끝났으면 **같은 슬러그·계약**으로 다시 기동한다(계약 파일 경로를 다시 전달).
  - 후속 작업이 **기존 어느 계약 범위에도 안 들어가면**(새 영역) → **P3로 돌아가** 오너십을 다시 배분한다(기존 계약 화이트리스트를 확장하거나 새 계약·에이전트를 추가). 확장·추가한 범위는 계약에 반영한다.
- **상태 업데이트(필수 — 재개 즉시, →C3·C4)**: 작업을 시작하기 전에 반드시 오케스트레이션 상태를 되돌려 놓는다.
  - 상태표: 해당 구현 에이전트 `완료 → 구현`(재분석이 필요하면 `분석`), **라운드 +1**, 갱신 일시.
  - **⛔ `## 현재 단계`가 `해결`(또는 `종료`)이면 활성 단계로 되돌린다** — `구현`(재분석 필요 시 `분석`). **재개하면 "해결"이 자동 해제**되는 것이다(안 되돌리면 목록의 "작업 상태"는 "해결됨"인데 칸반은 "구현"으로 어긋난다). `## 해결` 섹션이 있으면 재개 사실을 남기거나 비운다. 재통합(P7) 후 다시 `완료`가 되고, 필요하면 `dobby-resolve`로 재해결한다.
  - 이벤트 로그에 **`- {YYYY-MM-DD HH:MM} 후속: {슬러그} 피드백 재개 — {요약}`** 1줄(시각 HH:MM 필수).
- **규칙대로 진행**: 재개한 에이전트는 `dobby-impl`(비소스면 `dobby-produce`)로 **계약 허용 범위 안에서만** 작업한다. 그 뒤 **반드시 P5 코드리뷰(별도 리뷰 에이전트) → P6 피드백 루프**를 다시 거쳐 blocking=0을 만들고(리뷰 생략 금지 — →C5), 커밋·푸시(→C1) → **P7로 루트 브랜치에 재통합**하고 상태를 `완료`로 갱신한다(P7 규칙 그대로 — 통합 후 `dobby-explain` 갱신 포함). 필요하면 다시 `dobby-resolve`로 해결 표시.

## 확인 게이트 (좁음 — 이때만 사용자를 부른다)

1. **초기 계획 1회**: 팬아웃 K, 하위이슈/작업 키(문서 전용의 `TASK-{slug}` 포함), 범위 배분.
2. **순수 제품/디자인 결정**: 코드로 확정 불가한 것.
3. **최대 리뷰 라운드 초과**(기본 3).
4. **코드 외 작업**: 인프라·외부 콘솔 설정·배포 순서.
5. **메인 브랜치 밖 반영**(배포 등). — master PR·머지는 확인이 아니라 **금지**.

> **⛔ 커밋·푸시는 게이트가 아니다(→C1):** 자기·에이전트·루트 브랜치의 커밋·푸시·머지는 동의 없이 자동 진행한다. 멈춰서 묻지 말 것. 동의가 필요한 건 없고, 정식 배포 베이스(master) PR·머지는 확인이 아니라 **금지**다.

> **게이트 전 필수 규칙(docs 참조 §강화):** 사용자 결정이 필요해 보이면 **묻기 전에 관련 코드를 다시 상세 분석**하고(필요하면 docs 참조 — `$ORCHESTRATION_DOCS_ROOT/{repo}` 문서 → 없으면 `$ORCHESTRATION_REPOS_ROOT/{repo}` 실제 소스), 스스로 판단 불가한지 재확인한다. 그래도 결정이 필요할 때만 사용자에게 묻는다.

## 실행 모드

자기·에이전트·루트 브랜치까지는 어느 모드든 **커밋·푸시·머지가 사용자 동의 없이 자동** 진행된다(→C1). 오직 master(정식 배포 베이스) 반영만 수동/금지.

### 모드 A — 대화형 (Agent 도구 + SendMessage) · 기본
- 서브·리뷰 에이전트를 `Agent` 도구로 기동. 피드백 루프는 `SendMessage`로 같은 에이전트를 이어가 컨텍스트를 유지한다. 병렬 구현은 여러 `Agent` 호출을 한 메시지에 담아 동시에 띄운다.

### 모드 B — 자율 (Workflow)
- P4~P6을 `Workflow` 스크립트로 자동·병렬 실행한다. `mode=B` 선택 자체가 Workflow opt-in이다.
- (`light` 지정 시: P4 구현은 메인 인라인으로 수행되어 Workflow 대상에서 빠지고, Workflow는 리뷰(P5) 스폰에만 적용된다.)

## 오케스트레이션 메타 (K와 무관하게 항상 생성)

```
$ORCHESTRATION_META/{루트키}/
├── orchestration.md     # 관제탑 보드(에이전트 상태표·범위 배분·이벤트 로그)
├── agents/{슬러그}.md · review-agent.md   # 계약(불변)
├── reviews/round-{n}/{슬러그}.md
├── agent-logs.json      # {슬러그}→서브에이전트 대화 로그 경로(spawn 시점에 즉시 기록)
├── docs-refs.md         # 착수 docs 게이트 결과(먼저 읽을 문서 경로 / 히트 없음 — P1 최초, 차단 게이트)
├── decisions.md         # 자율 판단 기록(사용자에게 안 묻고 스스로 정한 결정 — append-only)
├── side-effects.md      # 사이드 이펙트 분석(구현 설계의 파급 다각도 점검 — P3, P4 진입 게이트)
├── test-guide.md        # 사용자 수동 확인 가이드(사이드이펙트 검증 TC — P7 통합 후)
└── logs/{슬러그}.jsonl   # (선택) 서브 대화 로그 사본(영속화 — 원본은 tmp라 사라질 수 있음)
```

- **⛔ 에이전트 상태 단일 정본(→C3, 가장 중요)**: 에이전트별 상태는 **오직 `orchestration.md`의 `## 에이전트 상태표` 한 곳**에서만 관리한다. **`status.md`에는 에이전트 상태표를 두지 않는다**(status.md는 현재 단계·메타·단계별 진행·워크트리·세션만). 대시보드도 orchestration.md를 정본으로 읽는다.
  - **고정 스키마(반드시 이 헤더·컬럼 순서 그대로)**:
    ```markdown
    ## 에이전트 상태표
    | 슬러그 | 이름 | 설명 | 상태 | 라운드 | 착수 | 갱신 |
    |--------|------|------|------|--------|------|------|
    | {슬러그} | {롤 이름} | {현재 작업 한 줄} | {대기|분석|구현|리뷰|완료} | {n} | {YYYY-MM-DD HH:MM} | {YYYY-MM-DD HH:MM} |
    ```
    - `이름`·`설명`·`착수`·`갱신` 컬럼은 **필수**(빼지 말 것). `착수`가 없거나 날짜만이면 대시보드가 정체(주의)를 감지하지 못한다.
    - **⛔ `이름` = 롤(사람이 읽는 표시명)**. 대시보드가 슬러그 대신 이 이름을 보여준다.
      - 롤: **`분석`**(dobby-start·Explore) · **`개발자`**(dobby-impl) · **`산출자`**(dobby-produce 비소스) · **`리뷰어`**(review-agent).
      - **한 에이전트가 여러 롤을 겸하면 `/`로 붙인다** — 인라인(light)은 착수·분석·구현을 다 하므로 **`분석/개발자`**. ⛔ **리뷰어는 절대 겸하지 않는다**(리뷰어≠구현자 원칙 — 리뷰는 항상 별도 `리뷰어` 행).
      - **같은 롤이 여럿(K≥2)** 이면 담당 영역/repo를 붙여 구분: **`개발자·FE`·`개발자·BE`·`개발자·결제`** 등(계약의 담당 영역에서 짧게).
    - **`슬러그` = 파일·로그용 안전 식별자**(영문 kebab 권장, `agents/{슬러그}.md`·`reviews/round-n/{슬러그}.md`·`agent-logs.json` 키에 그대로 쓰임). **`/`·공백·특수문자 금지**(이름의 `/`를 슬러그에 넣지 말 것 — 경로가 깨진다). 이름과 별개로 유지한다.
    - **⛔ `설명` = 현재 작업 한 줄(툴팁용)**. **이름을 지을 때 함께**, 그 시점에 이미 아는 정보로 **가볍게 한 줄** 적는다(예: `리워드 고시 중복 렌더 제거`, `구글 원탭 FE 패키지 구현`). **추가 조회·분석으로 토큰을 쓰지 말 것** — 슬러그를 짓듯 가볍게. 라운드가 바뀌어 작업 내용이 달라지면 그때 한 줄만 갱신(선택).
- **에이전트 상태값(5상태 — 각 에이전트는 자기 집합 안에서만 움직인다)**: 칸반 열 `대기 · 분석 · 구현 · 리뷰 · 완료`.
  - 구현(dobby-impl)·산출(dobby-produce): `대기 → 구현 → 완료`
  - 분석(dobby-start)·감사(Explore): `대기 → 분석 → 완료`
  - 리뷰(review-agent): `대기 → 리뷰 → 완료`
  - 인라인(light): `대기 → 분석 → 구현 → 완료`
  - ⛔ **구현 에이전트는 `리뷰`를 갖지 않는다.** 리뷰가 도는 동안에도 구현 에이전트는 `구현` 유지, `리뷰`는 **리뷰 에이전트만** 갖는다. 수정(피드백 반영)도 별도 상태 없이 `구현`이다. 통합되면 `완료`.
  - ⛔ **상태 칸 표기 제약(계약·필수)**: `orchestration.md`의 `## 에이전트 상태표` **상태 칸에는 정확히 `대기`·`분석`·`구현`·`리뷰`·`완료` 5개 중 하나만** 적는다. **볼드(`**`)·괄호·라운드 꼬리(`(round-3)`)·다른 단어·공백 변형·`~중` 접미어 등 어떤 꾸밈도 금지.** 라운드는 **`라운드` 칸에만** 적는다. (대시보드 칸반이 이 정확한 값으로 열을 매칭하므로, 어긋나면 그 에이전트가 보드에서 **조용히 사라진다.**)
- **⛔ 파일 갱신 방식(→C8)**: 상태 파일은 대시보드가 결정적으로 파싱하므로 **작게 유지**하고, 갱신 시 전체를 통독하지 않는다. 결정론적 갱신은 `dobby-lib.sh` 함수로 한다(config.md "공용 헬퍼" — 설정 절차에서 이미 source 됨):
  - 상태표 행 추가 → `dobby_agent_add`, 상태 전이 → `dobby_agent_state`, 이벤트 로그 → `dobby_event`, agent-logs → `dobby_log`, 현재 단계 → `dobby_phase`. 전부 **통독 없이 append·부분수정**이라 토큰이 거의 안 든다.
  - `status.md`는 **현재 상태 인덱스**로만 작게 유지한다 — 무거운 내용(분석·구현 요약·diff·로그)은 `analysis.md`·`implementation.md`/`produce.md`·`agent-logs.json` 등 **별도 파일**에 둔다.
  - 헬퍼를 못 쓰는 상황이면 수동 fallback: `## 이벤트 로그`는 `printf '%s\n' "- {일시} ..." >> orchestration.md`로 append, 상태표는 **표 구역만** 읽어 그 행만 수정. 매 전이마다 전체 통독 금지.
- **⛔ 상태 선(先)갱신 원칙(→C4, 전이별 상세)**: 전이별로:
  - **스폰 전/직후**: 새 에이전트 **행을 상태표에 추가**(분석 스폰이면 `분석`, 구현/산출 스폰이면 `구현`, 리뷰 스폰이면 `리뷰`). `agent-logs.json`에만 기록하지 말 것 — **상태표에 행이 없으면 대시보드 보드에 아예 안 보인다.** P1 분석 에이전트도, 단계 도중 늘어나는 에이전트도 예외 없이 스폰 즉시 등록한다.
  - **구현/산출 완료 → 리뷰**: **리뷰 에이전트를 `리뷰`로 스폰**하고 **구현 에이전트는 그대로 `구현` 유지**(리뷰 도는 동안에도). 리뷰·통합이 없는 **읽기전용 감사/조사는 산출물 확인 후 바로 `완료`**.
  - **리뷰 피드백 반영(수정)**: 구현 에이전트는 계속 `구현`(별도 수정 상태 없음). 재리뷰 시 리뷰 에이전트는 다시 `리뷰`.
  - **통합 직후**: 그 구현 에이전트를 `완료`로. 리뷰가 끝난 리뷰 에이전트도 `완료`.
  - **착수 시각 기록(→C4)**: `착수` 열은 `갱신`과 **별개 열**이다. 비활성에 갔다가 재개(예: `완료 → 구현`)하면 새 시각으로 갱신한다. 시:분이 없거나 비어 있으면 대시보드가 정체를 감지 못 하므로 날짜만 적지 말 것.
- **서브 로그 경로는 spawn 시점에 즉시 `agent-logs.json`에 기록한다**(완료 후 일괄 수집 X). 서브 에이전트를 띄우면 그 실행 결과의 `output_file`(대화 로그 경로)이 나오므로, 그 값을 `{슬러그}: "{경로}"`(또는 단계가 여럿이면 `{슬러그}: {"analysis": …, "impl": …}`)로 바로 써 넣는다. → 대시보드가 **실행 중부터** 그 로그를 콘솔로 실시간 표시할 수 있다.
  - **⛔ 리뷰 재라운드는 같은 `review-agent` 슬러그를 재사용한다(→C5).** `review-agent-r2` 같은 라운드별 새 슬러그를 만들지 않는다. 라운드가 여럿이면 같은 슬러그 아래 하위 키로 기록한다: `"review-agent": {"round-1": "…", "round-2": "…"}`. 구현 에이전트의 후속 라운드도 마찬가지로 같은 슬러그 아래 `impl_r2` 등 하위 키를 쓴다(새 슬러그 금지).
- **자기(오케스트레이터) 세션을 `status.md`에 남긴다**: `## 세션` 섹션에 `- **세션 ID**: {sessionId}` / `- **작업 경로**: {cwd}` 1줄씩. → 대시보드가 사용자가 직접 실행한 오더의 세션 기록 콘솔을 찾을 수 있다.
- (선택·영속화) tmp의 서브 `output_file`은 정리되면 사라지므로, 필요하면 `logs/{슬러그}.jsonl`로 복사해 두면 종료 후에도 기록 콘솔에서 재생할 수 있다.
- **이벤트 로그(대시보드 타임라인의 유일한 소스)**: Phase 전이·에이전트 착수/완료·리뷰 결과·충돌 판단 등 모든 타임라인 사건을 **오직 `## 이벤트 로그` 섹션 한 곳에만**, 한 줄씩 **`- {YYYY-MM-DD HH:MM} {사건} — {한 줄 설명}`** 형식으로 누적한다.
  - ⛔ **다른 `##` 섹션(감사 결과·P4 착수·마무리 등)에 타임라인을 흩뿌리지 말 것** — 대시보드는 `## 이벤트 로그` 섹션만 읽으므로 다른 섹션의 사건은 **타임라인에서 통째로 누락**된다.
  - ⛔ 각 줄은 **반드시 `- `로 시작**하고 **그 바로 뒤에 날짜**가 온다. `- 이벤트: {날짜}` 처럼 접두어를 붙이거나 `- ` 없이 `{날짜}`만 쓰면 **파서가 못 읽어 누락**된다.
  - **시각(HH:MM)까지 반드시 포함**(날짜만 적으면 누락 가능).
- **⛔ 자율 판단 기록(`decisions.md`)**: **사용자에게 물어볼 법한데 객관적 근거로 스스로 정한 결정**은 그때마다 `decisions.md`에 **한 블록씩 append**한다(기존 내용은 절대 다시 쓰지 말고 뒤에만 붙인다 — 토큰 절약). **→ 헬퍼 `dobby_append {루트키} decisions.md "{결정 블록}"`**(읽기 없이 뒤에만 붙임). 대시보드가 이걸 파싱해 카드로 보여준다.
  - **기록 대상**: 설계 선택, 활성경로/데드코드 판정, 파일 오너십·충돌 해소, 스펙·값 확정, 하위이슈 분해 방식 등 **"물어봤을 수도 있으나 근거로 자율 결정한 것"**. 관용적·사소한 선택(변수명 등)은 **기록하지 않는다**(노이즈 방지).
  - **작성 주체**: **오케스트레이터가 단독으로** append(병렬 경쟁 방지). 서브 에이전트(dobby-impl 등)가 자율 결정한 주요 사항은 자기 요약에 남기고, 오케스트레이터가 이 파일에 옮겨 정리한다.
  - **블록 형식(정확히 지킬 것 — 대시보드 파서가 이 구조로 카드를 만든다)**:
    ```markdown
    ## {YYYY-MM-DD HH:MM} · {단계}·{에이전트} — {무엇을 정했나(한 줄)}
    - **선택**: {스스로 정한 방향}
    - **이유**: {객관적 근거(코드·사실). 쉬운 말로}
    - **다른 옵션**:
      - {옵션 A} — {택하지 않은 이유}
      - {옵션 B} — {택하지 않은 이유}
    ```
    - 각 결정은 **반드시 `## `로 시작하는 한 블록**. 첫 줄은 `## {시각} · {맥락} — {제목}`. `다른 옵션`이 없으면 `- **다른 옵션**: 없음`.
    - 파일 맨 위 제목은 `# 자율 판단 기록` 한 줄(최초 생성 시). 이후 블록만 계속 append.
- **사이드 이펙트 분석(`side-effects.md`) — P3에서 작성(P4 게이트)**: 구현 설계의 파급을 **다각도로** 점검해 오케스트레이터가 1개로 정리한다. 대시보드가 `## ` 블록을 카드로 렌더한다.
  - **각도(항목마다 하나씩, 해당 없으면 "해당 없음"이라도 명시 → 다각도 확인 강제)**: ①기능 회귀(기존·인접 기능) ②국내/글로벌 이중 경로(static↔apps/global) ③공유 모듈·의존(공용 컴포넌트·widgets·패키지) ④데이터·상태·캐시 ⑤성능 ⑥보안·권한·인증 ⑦엣지 케이스·실패 경로 ⑧배포·롤백·마이그레이션.
  - **블록 형식**:
    ```markdown
    ## [영향도: 높음|중간|낮음] {각도} — {무엇이 파급될 수 있나(한 줄)}
    - **영향**: {구체적으로 무엇이 어떻게}
    - **가능성/심각도**: {높음/중간/낮음 + 근거}
    - **완화**: {어떻게 막거나 확인하나}
    ```
    - 파일 맨 위 제목 `# 사이드 이펙트 분석` 한 줄. 각 항목은 `## `로 시작하는 한 블록.
- **확인 가이드(`test-guide.md`) — P7 통합 후 작성**: 사용자가 **직접** 사이드이펙트를 확인하는 **수동 TC**를 오케스트레이터가 diff·implementation.md·side-effects.md 근거로 1개로 정리한다. 대시보드가 `## ` 블록을 카드로 렌더한다.
  - **블록 형식**(각 TC 한 블록. `[회귀]`는 사이드이펙트 확인, `[기능]`은 새 기능 확인 — **회귀 TC를 반드시 포함**):
    ```markdown
    ## TC-{n} · [회귀|기능] {화면/기능} — {무엇을 확인}
    - **화면/진입**: {URL·메뉴 경로}
    - **전제**: {국가·로그인·데이터}
    - **절차**:
      1. …
      2. …
    - **기대 결과**: {이래야 정상}
    - **왜(관련 위험)**: {이 변경이 이걸 건드릴 수 있어서}
    ```
    - 파일 맨 위 제목 `# 확인 가이드 (수동 TC)` 한 줄. 각 TC는 `## `로 시작하는 한 블록.
- **이 메타는 에이전트 수와 무관하게 항상 만든다(→C2).** 에이전트가 하나뿐이면 `orchestration.md` 보드는 1행, `agents/`에는 계약 1개(화이트리스트=전체 범위), `reviews/`에는 그 1개 에이전트의 라운드가 쌓이고, 루트 키가 곧 그 에이전트 키이므로 이 메타와 work 파일(analysis.md 등)이 같은 `$ORCHESTRATION_META/{루트키}/` 폴더에 함께 놓인다(에이전트가 여럿이면 work 파일은 각 하위 에이전트 키 폴더에).

## 승인 게이트

- **승인 불필요(자동, →C1)**: 하위/에이전트 브랜치와 상위(루트) 브랜치까지의 커밋·푸시·머지·origin 푸시.
- **⛔ 금지**: `$ORCHESTRATION_DEFAULT_BASE`로의 PR 생성·머지.
- **승인 필요**: 메인 브랜치 밖으로 나가는 반영(배포 등).

## 주의

- **사실 기반 원칙(최우선, →C7)**: 서브·리뷰 에이전트에게도 이 원칙을 전달한다.
- 어려운 용어를 지양하고 쉬운 말로 설명한다.
- 계약(불변)/`status.md`(진행)/`orchestration.md`(관제탑)의 역할을 섞지 않는다.
- 워크트리에는 `node_modules`가 없어 커밋 시 `--no-verify`로 훅을 건너뛴다.
