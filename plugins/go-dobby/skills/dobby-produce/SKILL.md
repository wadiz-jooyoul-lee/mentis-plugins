# dobby-produce

`dobby-order`(오케스트레이터)가 P4에서 기동한 **서브 에이전트의 산출 단계**를 담당하는 building block. `dobby-impl`의 비소스(non-code) 대응물이다. 코드 구현이 아니라 **문서·리서치·분석·설정 산출물** 등을 만든다.

> 이 스킬은 오케스트레이터가 P3에서 작업을 **`work-type = 비소스`**로 분류했을 때 `dobby-impl` 대신 호출된다. 코드 작업은 여전히 `dobby-impl` + `dobby-test`를 쓴다.

> **코드리뷰(내용 리뷰)는 이 스킬이 하지 않는다.** 검증·리뷰는 오케스트레이터(`dobby-order` P5)가 별도 리뷰 에이전트로 단독 수행한다(리뷰어≠산출자). 이 스킬은 산출 → 리뷰 요청(미커밋) → 피드백 반영 → 통과 후 (repo면) 커밋·푸시만 한다.

이 스킬을 실행하는 주체는 **서브 에이전트**다. 메인이 기동 시 **계약 파일 경로**(`$ORCHESTRATION_META/{루트키}/agents/{슬러그}.md`)를 전달한다 — 에이전트 수와 무관하게 항상 계약을 받는다(단일 에이전트면 계약 1개, 화이트리스트=전체 범위).

## 산출·조사 원칙 (필수 — 시작 전 숙지)

**`${CLAUDE_PLUGIN_ROOT}/reference/analysis-discipline.md`를 그대로 따른다.** 핵심:
1. **활성 경로 확정 먼저**: 코드 관련 조사·산출이면, 대상 기능이 실제로 실행되는 경로(앱/엔트리/라우트)를 증명한 뒤 근거로 삼는다. 국내 `static/entries/*`(레거시)와 `apps/global`(이관본) 이중화에 주의해 실행되는 쪽만 근거로 하고 안 되는 쪽은 데드코드로 명시. 확정 못 하면 멈추고 보고.
2. **전제 반증 허용**: 주어진 전제는 검증 대상이다. 사실로 확인되지 않으면 반박하고 코드/자료가 가리키는 결론을 제시한다.
3. **사실 기반·미증명 금지**: 모든 서술은 확인된 사실이어야 하고 근거(**`파일:라인`**·문서 출처)를 댄다. 추측은 추측이라 표기하고, 미증명 위에 결론을 세우지 않는다.

또한 **`${CLAUDE_PLUGIN_ROOT}/reference/role-personas.md`의 "비소스 산출 에이전트" 판단 기준**(출처 강제·모르면 "확인 필요"·완결성 자문)을 따른다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/go-dobby/config.env`를 source 해 환경 변수를 불러오고, **이미 값이 있는 변수는 묻지 않고** 빠진 값만 규칙대로 채워 저장·export 한다. 메타 루트 `ORCHESTRATION_META`, 변수 목록·기본값, 폴더 배치(워크트리 `$ORCHESTRATION_WORKSPACE/subtree/` · 메타 `$ORCHESTRATION_META/`)는 모두 그 문서에 있다. 이하 메타 경로는 `$ORCHESTRATION_META` 기준.

## 사전 조건

- **`dobby-start`의 산출물이 있어야 한다**: `$ORCHESTRATION_META/{키}/analysis.md`, `status.md`. 없으면 "먼저 dobby-start로 착수·분석하라"고 알리고 중단한다.
- 산출물이 repo에 들어가는 경우 status.md의 **워크트리 경로**가 존재해야 한다. repo 밖 산출(리포트 등)이면 워크트리가 없어도 된다.

## 산출물 위치 (두 모드 — analysis.md로 판별)

- **(a) repo 산출**: 산출물이 특정 repo의 파일(예: repo 내 문서 추가·수정)이면 → **워크트리에서 작업**하고, `dobby-impl`과 동일하게 **리뷰 통과 후** `commit --no-verify` → 자기 브랜치 푸시(B안 — 절차 4·5). 루트 브랜치로의 상위 통합은 dobby-order가 맡는다.
- **(b) meta 산출**: 산출물이 repo에 들어가지 않는 리포트/분석/조사 결과면 → **`$ORCHESTRATION_META/{키}/deliverables/`** 폴더에 저장한다(브랜치·커밋 없음).
- 어느 모드인지는 analysis.md의 작업 정의로 판별한다. 애매하면 메인/사용자에게 확인한다(임의 판단 금지).

## 타당성/개발 인계 형식 (produce 기본 산출 — 구현 가능 여부 판단)

`dobby-order`가 `produce`를 **목적 문장 없이** 준 경우(기본), 코드를 **읽기만** 해서 "구현 가능 여부"를 판단하는 리포트를 `deliverables/feasibility.md`에 아래 형식으로 만든다. **구현·수정은 하지 않는다.**

```markdown
# {이슈키} 구현 가능 여부 판단

## 결론
- 가능 / 조건부 가능 / 불가  (+ 한 줄 이유)

## 근거 (사실)
- {확인한 사실}  (`파일:라인`·문서 출처 필수)

## 필요한 변경 범위 (예상)
- {고쳐야 할 파일·영역 후보}

## 리스크·불확실성
- {막힐 수 있는 지점·미확인 항목}

## 개발 인계 (다음 개발이 재조사를 덜 하도록)
- **활성 경로(증명)**: {이 기능이 실제 실행되는 앱/엔트리/라우트}
- **변경 범위 후보(화이트리스트)**: {개발 시 손댈 파일 목록}
- **구현 방향 초안**: {어떻게 고치면 되는지}
- **남은 확인**: {개발 전에 추가로 확인할 것}

## 권고 (다음 단계)
- {개발 오더로 진행 시 `ref={이 키}`로 이 조사를 재활용하도록 안내}
```

- "개발 인계"는 이후 개발 오더가 `ref=`로 **그대로 재활용**한다(활성 경로·변경 범위·리스크를 다시 파지 않게). 근거 없는 단정 금지 — 확인 못 한 건 "남은 확인"에 남긴다.
- 목적 문장이 있으면(기획 보강·자료 조사 등) 그 산출물을 만들고, 이 타당성 형식은 쓰지 않는다.

## 절차

### 1. 컨텍스트 숙지 (산출 전)
- **`analysis.md`(작업 이해·계획)를 가장 먼저 읽는다.** 이 스킬은 여기 정의된 산출물을 만드는 것이 목적이다.
- **계약 파일**(`agents/{슬러그}.md`)의 **허용/금지 범위**를 확인한다(단일 에이전트면 화이트리스트가 전체 범위로 적혀 있다).
- 입력 문서(기획서·요청 문서 등)가 있으면 함께 숙지한다.
- 정의가 모호하거나 계약과 충돌하면 **임의 판단 말고** 메인/사용자에게 확인한다.

### 2. status.md 전환 + 산출 요약
- 현재 단계를 **`산출`**로, `갱신` 일시를 기록한다.(대시보드 칸반에선 `구현` 열로 접힌다.)
- **`$ORCHESTRATION_META/{키}/produce.md`**에 요약 스냅샷을 남긴다(산출 대상·구성·핵심 결정·근거 출처·미해결). 산출물 본문을 여기 통째로 옮기지 않는다(본문은 산출물 파일에).

### 3. 산출
- **허용 범위 안에서만** 산출한다. 범위 밖이 필요하면 임의로 건드리지 말고 메인에 요청한다.
- **docs 참조(필수)**: 확인이 필요한 사실은 → `$ORCHESTRATION_DOCS_ROOT/{repo}.md`(파일) 또는 `$ORCHESTRATION_DOCS_ROOT/{repo}/`(폴더) 문서 → 없으면 `$ORCHESTRATION_REPOS_ROOT/{repo}` 실제 소스를 읽어 확인하고 **반드시 사실 기반으로만** 산출한다. 추측 금지. (리서치·분석 산출물은 특히 근거를 `파일:라인`·출처로 명시한다.)
- **(a) repo 산출**: 워크트리의 대상 파일을 만든다/고친다.
- **(b) meta 산출**: `$ORCHESTRATION_META/{키}/deliverables/`(없으면 `mkdir -p`)에 산출물 파일을 저장한다. 파일명은 산출물 성격을 드러내게 한다.

### 4. 리뷰 요청 (커밋 전 — B안)
- **(a) repo 산출**: **⛔ 아직 커밋·푸시하지 않는다.** 변경을 워크트리에 **미커밋 상태로 둔 채** 메인(dobby-order)에 **내용 리뷰**를 요청한다(리뷰는 `git diff {fork}`·워크트리 파일을 본다).
- **(b) meta 산출**: `deliverables/`에 저장된 산출물 파일 자체를 리뷰 대상으로 메인에 요청한다(원래 커밋 없음). `produce.md`에 산출물 경로 기록.
- status.md 단계를 `리뷰`로 둔다. lint/build/브라우저 테스트는 **하지 않는다**(코드가 아님).

### 5. 피드백 반영 → 리뷰 통과 후 (repo 산출) 커밋·푸시 (라운드)
- 메인의 리뷰 에이전트가 남긴 피드백(`$ORCHESTRATION_META/{루트키}/reviews/round-{n}/{슬러그}.md`)을 받으면:
  - 단계를 `산출`로 되돌리고, **허용 범위 안에서** (repo면) 워크트리 / (meta면) `deliverables/` 파일에서 **그대로 반영**한다(아직 커밋 안 함). 반영을 `produce.md`에 기록 → 재리뷰 요청. **라운드 중 커밋·푸시하지 않는다.**
- blocking=0이 될 때까지 메인 주도로 `산출`↔`리뷰` 반복(최대 라운드는 dobby-order가 관리).
- **⛔ 리뷰 통과 후 — (repo 산출만) 처음이자 한 번, 커밋·푸시**: 워크트리에 `node_modules`가 없을 수 있어 **`git commit --no-verify`**. 메시지는 `docs:`/`chore:` 등 산출 성격 타입 + `{키}` + 실제 변경 요약. ⛔ 오케스트레이션 내부 용어(`round-N`·"리뷰 반영"·라운드/슬러그/메타 경로 등) 금지, 저장소 금지 서명(`Co-Authored-By`·"Generated with …") 금지. → **자기 브랜치에만 푸시**(베이스 자동 머지·PR 금지). `produce.md`에 커밋 해시 기록. **meta 산출은 커밋 없음**(deliverables 저장으로 끝).

## 내용 리뷰 기준 (dobby-order P5 리뷰 에이전트가 사용 — 참고)

코드 루브릭(A~E) 대신 **비소스 산출물 기준**으로 리뷰된다:
- **정확성**: 사실과 일치하는가(코드/문서 근거로 검증 가능한가). 추측·오류 없는가.
- **완결성**: 요청/analysis.md의 범위를 빠짐없이 다뤘는가.
- **요청 부합**: 기획서·요청 문서의 의도에 맞는가.
- **근거 명시**: 주장·수치에 출처(`파일:라인`·문서·조회)가 붙어 있는가.
- **규약**: 문서 컨벤션(있으면 그 repo의 문서 규칙)·용어(GLOSSARY) 준수.

## 주의

- **사실 기반 원칙(최우선)**: 산출·판단은 확인된 사실에만 근거한다. **추측 금지.** 확인 못 한 것은 `produce.md`에 "미확인"으로 남긴다.
- 이 스킬은 **자기 범위(워크트리 또는 자기 키 폴더) 안에서만** 동작한다. 다른 에이전트 영역·상위 브랜치를 건드리지 않는다.
- **내용 리뷰(검증)는 이 스킬이 하지 않는다** — 오케스트레이터(dobby-order P5)가 단독 수행한다. 브라우저 테스트(`dobby-test`)는 비소스 작업에 해당 없음.
- 어려운 용어를 지양한다.
