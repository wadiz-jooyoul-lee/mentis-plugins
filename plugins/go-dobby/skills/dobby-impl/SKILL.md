# dobby-impl

`dobby-order`(오케스트레이터)가 P4에서 기동한 **서브 에이전트의 구현 단계**를 담당하는 building block. 자기 워크트리·브랜치에서 계약이 정한 허용 범위 안에서만(단일 에이전트면 전체 범위) 구현·검증한 뒤, **커밋 전 미커밋 상태로** 메인에 리뷰를 요청하고, 리뷰 통과 후에 커밋·푸시한다(B안).

> **코드리뷰는 이 스킬이 하지 않는다.** 리뷰는 오케스트레이터(`dobby-order` P5)가 **별도 리뷰 에이전트**로 단독 수행한다(리뷰어≠구현자, 범위 준수 검증 포함). 이 스킬은 구현 → 리뷰 요청(미커밋) → 피드백 반영 → 통과 후 커밋·푸시만 한다. (예전 `issue-impl`의 셀프리뷰는 오케스트레이터 리뷰와 중복이므로 제거했다.)

이 스킬을 실행하는 주체는 **서브 에이전트**다. 메인이 기동 시 **계약 파일 경로**(`$ORCHESTRATION_META/{루트키}/agents/{슬러그}.md`)를 전달한다 — 에이전트 수와 무관하게 항상 계약을 받는다(단일 에이전트면 계약 1개, 화이트리스트=전체 범위).

## 구현 원칙 (필수 — 시작 전 숙지)

**`${CLAUDE_PLUGIN_ROOT}/reference/analysis-discipline.md`를 그대로 따른다.** 핵심:
1. **활성 경로 확정 먼저**: 고치려는 코드가 실제로 실행되는 경로인지 **라우트 등록·마운트로 증명**한 뒤 수정한다. 국내는 `static/entries/*`(레거시)와 `apps/global`(이관본)로 이중화돼 동일 기능이 양쪽에 있으니, **실행되지 않는 쪽을 고치는 실수**를 막기 위해 활성 경로를 먼저 확정한다. 확정 못 하면 멈추고 보고.
2. **전제 반증 허용**: 계약·분석이 준 전제도 활성 코드로 재확인한다. 어긋나면 반박하고 사실을 우선한다.
3. **사실 기반·미증명 금지**: 수정 근거를 **`파일:라인`**으로 대고, 미증명 메커니즘 위에 구현하지 않는다(미증명이면 확인 방법과 함께 보고).

또한 **`${CLAUDE_PLUGIN_ROOT}/reference/role-personas.md`의 "구현 에이전트" 판단 기준**(요구 의심·미래 독자 기준·실패 경로 동시 처리·최소 코드·정밀한 수정)을 따른다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/go-dobby/config.env`를 source 해 환경 변수를 불러오고, **이미 값이 있는 변수는 묻지 않고** 빠진 값만 규칙대로 채워 저장·export 한다. 메타 루트 `ORCHESTRATION_META`, 변수 목록·기본값, 폴더 배치(워크트리 `$ORCHESTRATION_WORKSPACE/subtree/` · 메타 `$ORCHESTRATION_META/`)는 모두 그 문서에 있다. 이하 메타 경로는 `$ORCHESTRATION_META` 기준.

## 사전 조건

- **`dobby-start`의 산출물이 있어야 한다**: `$ORCHESTRATION_META/{키}/analysis.md`, `status.md`. 없으면 "먼저 dobby-start로 착수·분석하라"고 알리고 중단한다.
- status.md의 **워크트리 경로**가 실제로 존재해야 한다. 없으면(정리됨) 사용자/메인에 알리고 재생성 여부를 확인한다.

## 절차

### 1. 컨텍스트 숙지 (구현 전)
- **`analysis.md`(원인·수정 설계)를 가장 먼저 읽는다.** 이 스킬은 여기 적힌 설계를 구현하는 것이 목적이다.
- **계약 파일**(`agents/{슬러그}.md`)의 **수정 허용/금지 범위**를 확인한다(단일 에이전트면 화이트리스트가 전체 범위로 적혀 있다).
- 워크트리 기준 코딩 규약 문서를 확인한다: 루트 및 대상 영역의 `CLAUDE.md`, `NAMING_CONVENTIONS.md`, `GLOSSARY.md`. (이는 규약대로 구현하기 위한 것 — 리뷰 자체는 dobby-order가 한다.)
- 설계가 모호하거나 계약과 충돌하면 **임의 판단 말고** 메인/사용자에게 확인한다.

### 2. status.md 전환 + 구현 요약
- 현재 단계를 **`구현`**으로, `갱신` 일시를 기록한다.
- **`$ORCHESTRATION_META/{키}/implementation.md`**에 요약 스냅샷을 남긴다(건드린 파일 경로·핵심 설계 결정·경계/공용 파일 처리·미해결). 파일별 장황한 diff는 옮기지 않는다(실제 내역은 이 에이전트 대화 로그에 남으며 메인이 수집).

### 3. 구현
- **허용 범위 안에서만** 구현한다. 범위 밖이 필요하면 임의로 건드리지 말고 메인에 요청한다(충돌 방지의 핵심). 기존 코드 패턴·`CLAUDE.md` 규약을 따르고, 요청되지 않은 리팩터링·기능 추가는 하지 않는다.
- **docs 참조(필수)**: 애매하거나 확인이 필요한데 현재 코드로 확인 불가하면 → `$ORCHESTRATION_DOCS_ROOT/{repo}.md`(파일) 또는 `$ORCHESTRATION_DOCS_ROOT/{repo}/`(폴더) 문서 → 없으면 `$ORCHESTRATION_REPOS_ROOT/{repo}` 실제 소스를 읽어 확인하고 **반드시 사실 기반으로만** 구현한다. 추측 금지.

### 4. 검증 + 리뷰 요청 (커밋 전 — B안)
- 검증 명령(lint/type/build)이 영역 `CLAUDE.md`에 있으면 워크트리에서 실행해 통과를 확인한다. 실패는 고치고, 못 고치면 `implementation.md`에 기록·보고한다.
- **⛔ 아직 커밋·푸시하지 않는다.** 변경을 **워크트리에 미커밋 상태로 둔 채** 메인(dobby-order)에 **리뷰를 요청하고 대기**한다 — 코드리뷰는 이 미커밋 변경(`git diff {fork}`, `fork`=`git merge-base origin/{base} HEAD`)을 대상으로 한다. status.md 단계를 `리뷰`로 둔다. 메인 지시 없이 상위 브랜치·다른 에이전트 범위를 건드리지 않는다.
- `implementation.md`에 구현 요약(변경 파일·요지)을 기록한다(커밋은 아직 없음).

### 5. 피드백 반영 → 리뷰 통과 후 커밋·푸시 (라운드)
- 메인의 리뷰 에이전트가 남긴 피드백(`$ORCHESTRATION_META/{루트키}/reviews/round-{n}/{슬러그}.md`)을 받으면:
  - 단계를 `구현`으로 되돌리고(수정=구현), **허용 범위 안에서 워크트리에서 그대로 반영**한다(아직 커밋 안 함). 명백한 것(규약 위반·오류·nit)은 바로, 설계 변경/해석 갈림은 메인/사용자 확인 후.
  - 반영을 `implementation.md`에 기록하고 → 메인에 **재리뷰 요청**(단계 `리뷰`). **라운드 중에는 커밋·푸시하지 않는다.**
- blocking(blocker·major)=0이 될 때까지 메인 주도로 `구현`↔`리뷰`를 반복한다(최대 라운드는 dobby-order가 관리). 범위 밖 수정·계약 상충 지적은 메인에 확인 후 처리한다.
- **⛔ 리뷰 통과(blocking=0) 후 — 처음이자 한 번, 커밋·푸시**:
  - 워크트리에는 `node_modules`가 없어 pre-commit 훅이 실패할 수 있으므로 **`git commit --no-verify`**로 리뷰 통과분 전체를 **의미 단위 커밋**한다.
  - **커밋 메시지 규칙**: 형식 `fix: {키} ...`(버그)/`feat: {키} ...` — 접두어 뒤엔 **실제 코드 변경(무엇을·왜)** 을 간결히. ⛔ **오케스트레이션 내부 용어 금지**(`round-3`·"리뷰 반영"·라운드/슬러그/`reviews/`·메타 경로 등 — 그 수정이 코드에 한 일로 쓴다, 예 `fix: {키} 로그인 실패 시 null 체크 추가`). ⛔ **금지 서명 제거**(`Co-Authored-By`·"Generated with …" 등, 저장소 규약이 금지하면 기본값 포함 제거; 불명확하면 그 repo `CLAUDE.md`·`.github` CI 규칙 확인).
  - **자기 브랜치(`{prefix}/{키}`)에만 푸시한다.** ⛔ 베이스 브랜치로의 **자동 머지·PR 생성은 하지 않는다**(상위 통합은 dobby-order, 최종 master 반영은 사용자 수동 PR).
  - `implementation.md`에 커밋 해시·푸시 브랜치를 기록한다.

## 주의

- **사실 기반 원칙(최우선)**: 구현·판단은 확인된 사실에만 근거한다. **추측 금지.** 확인 못 한 동작은 코드/docs로 확인하고, 불가하면 `implementation.md`에 "미확인"으로 남긴다.
- 이 스킬은 **자기 워크트리·브랜치 안에서만** 동작한다. 다른 에이전트 영역·상위 브랜치를 건드리지 않는다.
- 커밋·푸시는 허용되나 **PR 생성·베이스 머지는 하지 않는다.**
- **코드리뷰는 이 스킬이 하지 않는다** — 오케스트레이터(dobby-order P5)가 별도 리뷰 에이전트로 단독 수행한다. 이 스킬은 그 피드백을 반영할 뿐이다.
- **후속(완료 후 피드백) 재개**: 통합·해결 이후 같은 슬러그·계약으로 다시 기동될 수 있다(dobby-order P8). 이때도 새로 시작하지 않고 **status.md 단계를 `구현`(재분석 필요 시 `분석`)으로 되돌리고**, 위 4·5단계(구현→미커밋 리뷰 루프→통과 후 커밋·푸시)를 **동일하게** 밟는다. 상태 갱신을 빼먹지 않는다(오케스트레이션 보드 갱신은 메인 dobby-order가 담당).
- 어려운 용어를 지양한다.
