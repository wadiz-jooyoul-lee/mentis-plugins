# dobby-resolve

이슈/작업을 **"해결"로 표시**하는 비파괴 스킬. 상태만 바꾸고 **아무것도 삭제하지 않는다.** "테스트 통과"와 "해결로 선언"을 분리해, 통과했더라도 추가 수정 여지를 남긴다.

> 워크트리 실제 제거(정리)는 이 스킬이 아니라 **`dobby-end`**가 하며, 사용자가 직접 실행한다. `dobby-resolve`는 상태 표시까지만이다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/go-dobby/config.env`를 source 해 환경 변수를 **읽기만** 한다. ⛔ **이 스킬은 config.env를 저장·수정·생성하지 않는다**(값 변경은 `dobby-init` 전용 — config.md '비파괴 원칙'). config.env 파일이 아예 없으면 멈추고 `/dobby-init`을 먼저 실행하도록 안내한다. 메타 루트 `ORCHESTRATION_META`, 변수 목록·기본값, 폴더 배치(워크트리 `$ORCHESTRATION_WORKSPACE/subtree/` · 메타 `$ORCHESTRATION_META/`)는 모두 그 문서에 있다. 이하 메타 경로는 `$ORCHESTRATION_META` 기준.

## 입력

- `args`: `{키}`(이슈 키 또는 작업 키 `TASK-...`). 없으면 현재 브랜치에서 키를 추출하거나 물어본다.
- `undo`(선택): `{키} undo`면 **해결 표시를 무르는 모드**(아래 "취소(undo)"). 기본은 해결 표시.

## 사전 조건

- `$ORCHESTRATION_META/{키}/status.md`가 있어야 한다. 없으면 "먼저 dobby-order로 진행하라"고 안내하고 중단한다.

## 절차

### 1. 상태·근거 확인 (사실 기반)
- `status.md`의 현재 단계와 `단계별 진행`·`테스트 실행 이력`을 읽는다.
- **해결 근거를 확인한다**:
  - 최신 `$ORCHESTRATION_META/{키}/test-runs/{시각}/result.md`(있으면)의 집계 — **전부 PASS 권장**. 실패/중단이 있으면 그대로 해결로 올리지 말고 사용자에게 확인한다.
  - 리뷰 클린 여부(`reviews/`) 및 루트 브랜치 통합 완료(에이전트가 여럿이면 각 브랜치가 루트로 머지됨; 하나면 자기 브랜치가 곧 루트).
- 테스트 결과가 없어도 **사용자가 명시적으로 해결로 표시하라고 하면** 그 사실을 근거로 기록하고 진행한다(근거에 "사용자 지정"으로 남김).

### 2. 상태 표시 (비파괴)
- `status.md`의 **현재 단계 → `해결`**, `담당 스킬 → dobby-resolve`, 갱신 일시 기록.
- **에이전트 상태 정리(안전망)**: `status.md`·`orchestration.md`의 에이전트 표에서 아직 `완료`가 아닌 에이전트(예: `구현`·`리뷰`)를 **`완료`로 맞춘다**(해결 = 통합까지 끝났다는 뜻). dobby-order P7에서 이미 `완료`면 그대로 둔다. (이 정리를 빼면 대시보드 보드가 해결 이후에도 `구현`/`리뷰`로 계속 보인다.)
- `## 해결` 섹션에 근거를 남긴다:
  ```markdown
  ## 해결
  - **처리 일시**: {YYYY-MM-DD HH:MM}
  - **근거**: 리뷰 클린({round}) · 테스트 결과 `test-runs/{시각}/`(집계 성공X/실패0/skip Z) · 통합 브랜치 {feature/루트키 또는 자기 브랜치}
  - **비고**: {추가 수정 여지 등}
  ```
- **삭제·제거는 하지 않는다**: 워크트리(`$ORCHESTRATION_WORKSPACE/subtree/{repo}-{키}`)와 메타 폴더(`$ORCHESTRATION_META/{키}/`)를 그대로 유지한다.
- **Jira 상태는 건드리지 않는다**(메타만 갱신). 실제 이슈 상태 전환이 필요하면 사용자가 직접 한다.

### 3. 안내
- 해결로 표시했음을 요약해 보여준다(근거 포함).
- **추가 수정(사용자 피드백 후속)이 필요하면** 폴더·워크트리가 살아 있으므로 이어서 작업할 수 있음을 안내한다. 이때는 `dobby-order`의 **P8(후속 작업 재개)** 규칙을 따른다 — **그 범위를 소유한 담당 에이전트가** 이어받고, 재개 즉시 오케스트레이션 상태(에이전트 상태 `구현`·라운드+1·이벤트 로그)를 되돌린 뒤 리뷰·통합 루프를 동일하게 거친다. 임의로 상태 갱신 없이 고치지 않는다.
- 워크트리를 실제로 정리하려면 **사용자가 `dobby-end`를 직접 실행**하면 됨을 안내한다.

## 취소 (undo) — `/dobby-resolve {키} undo`

해결 표시를 **선언만 되돌린다**(비파괴). 코드·워크트리·브랜치·Jira는 건드리지 않는다.
- `status.md`의 **현재 단계 `해결` → `통합`**(작업이 끝난 직전 단계)로 되돌리고, 갱신 일시 기록. 목록 "작업 상태"가 다시 "작업중"으로 돌아온다.
- **`## 해결` 섹션을 제거**한다(또는 "취소됨 {일시}" 한 줄로 대체).
- **에이전트 상태는 그대로 둔다**(`완료`) — 이미 통합된 사실은 유효하다. 이건 "재작업"이 아니라 **해결 선언만 무르기**다. (더 고쳐야 하면 `dobby-order` P8 재개 흐름으로 진행 — 그건 에이전트를 `구현`으로 되돌린다.)
- `orchestration.md` 이벤트 로그에 **`- {YYYY-MM-DD HH:MM} 해결 취소 — 해결 표시 해제`** 1줄.

## 주의

- **비파괴 원칙**: 이 스킬은 상태 표시만 한다. 워크트리·메타·브랜치·Jira를 삭제·변경하지 않는다.
- **사실 기반**: 해결 근거는 확인된 사실(테스트 결과·리뷰·통합)로만 기록한다. 실패가 있으면 임의로 해결로 올리지 않고 사용자에게 확인한다.
- 어려운 용어를 지양한다.
