# dobby-explain

구현이 끝난 오더를 **비전공자도 이해할 수 있는 "구현 내용" 문서로 정리**하는 비파괴 스킬. `$ORCHESTRATION_META/{키}/explainer.md` **한 파일만** 만든다(다른 것은 건드리지 않음).

> 핵심 원칙: **실제 구현한 내용만** 설명한다. 일반론·개념 나열이 아니라, "이 오더에서 실제로 무엇을 만들었고 어떻게 동작하는가"를 산출물 근거로 쓴다. 확인 안 된 내용은 지어내지 않는다.

## 설정 (첫 실행 시 확인)
작업을 시작하기 전에 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/go-dobby/config.env`를 source 해 환경 변수를 불러온다. 이하 메타 경로는 `$ORCHESTRATION_META` 기준.

## 사용법
`/dobby-explain {키} [regen]` — {키}는 이슈 키 또는 `TASK-{slug}`. 대상 폴더는 `$ORCHESTRATION_META/{키}/`.
- **인자 없음(기본)**: `explainer.md`가 없으면 전체 생성, **있으면 후속 변경분만 `## 후속(P8)` 섹션으로 append**(전체 재작성·전체 재읽기 안 함 — 토큰 절약).
- **`regen`**: 사용자가 명시적으로 재생성을 요청한 경우에만 **전체 산출물을 다시 읽어 처음부터 덮어쓰기**(매끈한 단일 문서로 재정리).

## 입력(근거) — 있는 것만 읽는다
1. `orchestration.md` — **교차 저장소 API 계약·확정된 결정·에이전트 역할표**. 흐름(요청/응답·status 값)의 뼈대. **가장 중요**.
2. `implementation.md` / `produce.md` — 구현/산출 요약(무엇을 만들었나).
3. `analysis.md` — 배경·원인·설계(왜 만들었나).
4. `reviews/round-*/*.md` — 품질·수정 이력(무엇을 보강했나).
5. `agent-logs.json` → 각 에이전트 대화 로그 — 실제 수정 파일·커밋·요약(구현 사실). 로그가 tmp라 없으면 생략.
6. Jira 이슈 제목·설명(있으면) — 한 줄 요약용.

> 근거가 부족하면 **코드/문서를 직접 더 읽어 사실을 확인**한다(추측 금지). 그래도 불명확하면 그 부분은 단정하지 말고 "미확인"으로 표시한다.

## 산출: `$ORCHESTRATION_META/{키}/explainer.md`
아래 구조로 **한국어·비전공자 톤**으로 쓴다. 전문용어는 처음 쓸 때 괄호로 풀이한다.

```markdown
# {기능명} — 구현 내용
> 이 오더에서 실제로 구현한 내용을 코드 지식 없이도 이해하도록 정리 (근거: orchestration.md 등)

## 한 줄 요약
{무엇을 만들었는지 한 문장}

## {개념} 이/가 뭔가요?   ← 필요할 때만(생소한 개념이면)
{짧고 쉬운 설명}

## 누가 무엇을 했나 (역할 분담)
| 주체 | 이번에 실제로 한 일 | 쉽게 말하면 |
|---|---|---|
| {브라우저/외부 서비스} | … | … |
| 프론트엔드(FE) | … | … |
| 백엔드(BE) | … | … |
(주체는 실제 구현 구성에 맞게 — 단일 repo면 FE/BE 구분 없이 "구현" 한 줄일 수도 있다)

## {각 주체}가 만든 것   ← 실제 파일·API·동작 기반으로 쉽게
…

## 주요 흐름
### {시나리오명}
​```mermaid
sequenceDiagram
  참가자간 실제 요청/응답을 단계로   ← orchestration.md의 API 계약·status 값 기반
​```
{그림 아래 1~2줄 쉬운 설명}
(로그인/가입/조회 등 실제 흐름마다 하나씩. 분기·조건은 flowchart로도 가능)

## 용어 풀이
- {용어}: {쉬운 뜻}

## 남은 일 (있으면)
- {미해결·후속(배포/검증 등)}
```

### 다이어그램 규칙
- 흐름은 **mermaid**로 그린다(`sequenceDiagram` 요청/응답, `flowchart` 분기·조건).
- 다이어그램의 단계·분기는 **실제 API 계약·상태값·코드 동작**에서 가져온다(예: `POST /경로`, `status: A|B`). 임의로 지어내지 않는다.
- 노드 라벨은 한국어로 쉽게. 참가자는 사용자·브라우저/외부·FE·BE 등 실제 구성대로.
- **⛔ mermaid 문법 안전 규칙(어기면 대시보드에서 "Syntax error"로 다이어그램이 안 그려진다)**:
  - **flowchart 노드 라벨은 항상 큰따옴표로 감싼다** — `A["텍스트"]`, `B{"텍스트"}`. 라벨에 **괄호 `( )`·콜론 `:`·`<br/>`·`→`·`/`·`#`·따옴표** 등이 있으면 따옴표 없이는 파싱이 깨진다. (예: `F["이전(외부)로 이탈<br/>정상"]`)
  - **점선/텍스트 링크는 정식 문법으로**: 텍스트가 붙은 링크는 `A -->|텍스트| B`, 점선은 `A -.->|텍스트| B`. `A -.텍스트.-> B`처럼 쓰지 않는다.
  - `sequenceDiagram` 메시지의 `<br/>` 줄바꿈은 허용되지만, `flowchart`는 위처럼 **라벨을 따옴표로** 감싸야 안전하다.
  - 화살표 텍스트·참가자 별칭에 특수문자를 넣지 말고, 넣어야 하면 라벨을 따옴표로 감싼다.

### 톤·품질
- 비전공자 기준. 어려운 말은 풀이. 문장은 짧게.
- **사실 기반**: 산출물에 있는 것만. 모르면 단정하지 않는다.
- 코드 diff의 세부 구현(변수명 등)은 나열하지 말고, **동작·목적** 중심으로.

## 갱신 방식 (토큰 절약 — 기본 append, 재생성은 명시적일 때만)
- **`explainer.md`가 없으면** → 전체 생성(입력 근거를 읽어 처음 1회 작성).
- **이미 있으면(기본)** → **후속 작업으로 바뀐 부분만** 읽어(전체 소스 재읽기 금지) 문서 **맨 끝에 `## 후속(P8) {YYYY-MM-DD} — {요약}` 섹션을 append**한다. 기존 본문·다이어그램은 다시 쓰지 않는다. → P8마다 델타만 붙어 재읽기·재생성 비용이 안 든다.
  - 후속 섹션에도 사실 기반·비전공자 톤은 동일. 필요하면 그 섹션 안에 작은 mermaid를 새로 넣되 기존 다이어그램은 건드리지 않는다.
- **`regen` 인자를 받은 경우에만** → 전체 산출물을 다시 읽어 **처음부터 덮어쓰기**(누적된 후속 섹션들을 하나의 매끈한 문서로 재정리). 사용자가 명시적으로 요청할 때만.

## 비파괴
- `explainer.md`만 생성/갱신한다. 워크트리·다른 메타·Jira·코드는 건드리지 않는다.

## 자동 실행
- `dobby-order`가 통합(P7) 완료 후 이 스킬을 호출해 자동 생성한다(아래 "생명주기").
- 사용자가 대시보드 "구현 내용" 페이지의 생성 버튼으로도 실행한다(`/dobby-explain {키}`).
