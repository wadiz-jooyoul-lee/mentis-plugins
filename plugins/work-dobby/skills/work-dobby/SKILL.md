# work-dobby

상위 이슈를 하위이슈들로 쪼개, 각 하위이슈를 독립 브랜치·워크트리에서 서브 에이전트로 구현하고, 리뷰 루프를 거쳐 상위 브랜치로 통합하는 **오케스트레이터**. 이 스킬을 실행하는 주체가 **메인 에이전트**다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 `~/.config/work-dobby/config.env`를 읽어 환경 변수를 불러온다(`[ -f ~/.config/work-dobby/config.env ] && source ~/.config/work-dobby/config.env`). **이미 값이 있는 변수는 묻지 않는다.** 빠진 변수만 아래 규칙으로 채운다.

- **기본값이 있는 변수**: 현재 기본값을 보여주고 그대로 쓸지 바꿀지 물어본다.
- **선택 변수**(생략 가능): 생략 시 어떤 영향이 있는지 설명하고 생략을 허용한다.
- 사용자가 정한 값은 설정 파일에 저장하고(`mkdir -p ~/.config/work-dobby` 후 기록) `export` 한다.

| 변수 | 뜻 | 기본값 |
|------|-----|--------|
| `JIRA_BASE_URL` | Jira 사이트 주소 | `https://wadiz.atlassian.net` |
| `DOBBY_WORKSPACE` | 작업 루트(하위에 `subtree/`·`meta/`) | `$HOME/work/dobby-workspace` |
| `DOBBY_DEFAULT_BASE` | 기본 베이스 브랜치 | `master` |
| `DOBBY_REPOS_ROOT` | 원본 소스 저장소들이 있는 루트 | `$HOME/work` |
| `DOBBY_ENV_MAP` | 테스트 환경→호스트 매핑 | `dev=dev.wadiz.io,rc=rc.wadiz.kr,rc2=rc2.wadiz.kr,rc4=rc4.wadiz.io` |
| `DOBBY_DOCS_ROOT` | 참고 문서 루트(선택) | (없음 → 참고 문서 없이 진행) |
| `TEST_LOGIN_ID` / `TEST_LOGIN_PW` | 테스트 계정(선택) | (없음 → 로그인 필요 테스트는 건너뜀) |

워크트리(작업 폴더)는 `$DOBBY_WORKSPACE/subtree/`, 메타 파일은 `$DOBBY_WORKSPACE/meta/`에 둔다.

## 스킬 3종 관계 (building block)

```
work-dobby   ← 오케스트레이터(메인 에이전트). 전체 흐름을 지휘.
  ├─ agent-start  ← 하위이슈별 셋업+분석. 메인이 P1에서 하위이슈마다 실행.
  └─ agent-impl   ← 하위이슈 구현. 각 서브 에이전트가 P4에서 스스로 실행.
```

`work-dobby`는 새 로직을 만들기보다 위 두 building block과 `agent-start`가 정의한 메타 계층(계약·보드)을 **엮어서 지휘**한다.

## 입력

- `work-dobby {상위이슈키|URL} [base={브랜치}] [subtasks=키,키,...] [mode=A|B]`
  - **상위이슈**: 에픽/스토리 키 또는 Jira URL. 하위이슈를 `parent = {상위}` JQL로 자동 탐색하고 사용자에게 확인받는다.
  - `base=`: 상위 브랜치의 베이스(미지정 시 `$DOBBY_DEFAULT_BASE`, 기본값 master). 여기서 상위 브랜치 `{prefix}/{상위키}`를 만든다.
  - `subtasks=`: 대상 하위이슈를 명시(일부만 돌릴 때). 미지정 시 자동 탐색 결과 전체.
  - `mode=`: 실행 모드 A(대화형) / B(자율). 미지정 시 **A**.

## 절차 (Phase)

### P0. 베이스 셋업
- 소스 루트에서 `git fetch origin {base}` → `{base}` 최신화.
- 상위 브랜치·워크트리 생성: `git worktree add -b {prefix}/{상위키} $DOBBY_WORKSPACE/subtree/{repo}-{상위키} origin/{base}`.
- origin에 상위 브랜치 푸시(하위 워크트리가 원격 기준으로도 브랜치 가능하도록). **승인 불필요**(아래 "승인 게이트").
- 메타 루트 생성: `$DOBBY_WORKSPACE/meta/.agent-start/{상위키}/`(+ `agents/`, `reviews/`).

### P1. 하위이슈 착수·분석
- 각 하위이슈에 `agent-start {하위키} base={prefix}/{상위키}` 실행 → Jira "진행 중" 전환·워크트리 생성·코드 분석·`status.md`·테스트 목록 확보.

### P2. 분석 검증
- agent-start 산출물(분석)을 숙지하고, **기획서·Jira 이슈·(있으면) Figma** 와 맞는지 교차 검증한다.
- Figma 링크가 없으면 사실대로 표기하고 기획·Jira 기준으로 검증한다(추측 금지).
- 분석이 요구사항과 어긋나면 해당 하위이슈의 `agent-start` 분석을 보정한 뒤 다음으로 넘어간다.

### P3. 범위 배분 + 실행 모드 결정 (충돌 방지 핵심)
- **구현에 필요한 코드를 먼저 분석**해 하위이슈(에이전트)별 **수정 파일 화이트리스트**를 정한다.
- 파일 오너십이 겹치지 않으면 → **병렬로 바로 P4에 착수한다(사용자에게 묻지 않는다).**
- 겹치면 → **순차**: A 구현·리뷰·머지 완료 후, 갱신된 상위 브랜치에서 B 워크트리를 재생성(또는 리베이스)해 B 진행. 사유를 기록한다.
- 판단에 필요한 사실(예: 백엔드 스키마 passthrough 여부, 국내/글로벌 렌더 경로, API 계약 등)이 불확실하면 **묻지 말고 백그라운드 에이전트로 직접 확인해 판단한다.** 확인 결과가 범위/계약에 영향을 주면 화이트리스트·데이터 계약에 반영한다. 확인으로도 결론이 안 서거나 프론트 밖(백엔드/디자인) 작업이 필요할 때만 사용자에게 알린다.
- `agent-start`의 규격대로 `orchestration.md`(보드)·`agents/{슬러그}.md`(서브 계약)·`agents/review-agent.md`(리뷰 계약)를 작성한다. 계약의 "수정 허용 범위"에 위 화이트리스트를 채우고, 공유 데이터 계약(신규 필드명·구조)을 확정해 양쪽 계약서에 동일하게 명시한다.

### P4. 구현
- 각 서브 에이전트를 기동한다(실행 모드는 A/B, 아래 "실행 모드" 참조). **계약 파일 경로를 첫 지시로 전달**한다.
- 서브 에이전트는 `agent-impl`로 구현 → `status.md` `구현중`·`## 구현` 기록 → `commit --no-verify` → 자기 브랜치 푸시 → 메인에 리뷰 요청·대기.

### P5. 코드리뷰
- 리뷰 에이전트를 기동해 각 브랜치 diff를 리뷰한다(브랜치별 병렬 가능). 계약의 리뷰 기준(요구사항 충족·correctness·컨벤션·**범위 준수 검증**)을 따른다.
- 결과를 `reviews/round-{n}/{슬러그}.md`에 심각도·`파일:라인`·수정 제안으로 남긴다.

### P6. 피드백 루프
- 리뷰 결과를 해당 서브 에이전트에 전달한다(모드 A는 `SendMessage`로 같은 에이전트를 이어감). 서브 에이전트는 `agent-impl` 6단계로 반영·재푸시.
- **모든 수정 라운드 후에는 반드시 재리뷰한다(생략 금지).** 사용자·디자인 반영 등 리뷰 지적이 아닌 수정이라도, 코드가 바뀌었으면 `수정중 → 리뷰중`으로 재리뷰를 거친다. "작은 수정이라 생략"은 허용하지 않는다. 재리뷰는 그 라운드의 delta(직전 리뷰 이후 변경)를 대상으로 한다.
- blocking 이슈 0이 될 때만 `재통합대기`로 넘어간다. blocking 이슈 0이 될 때까지 `리뷰중`↔`수정중` 반복. **최대 라운드 기본 3**, 초과 시 사용자에게 에스컬레이션한다.
- 라운드·상태를 `orchestration.md`에 갱신한다.

### P7. 통합
- 리뷰 클린 → 상위 브랜치로 각 하위 브랜치를 **순차 머지** → 상위 브랜치 커밋·푸시. **승인 불필요**.
- **⛔ `$DOBBY_DEFAULT_BASE`(정식 배포 베이스)로의 PR 생성·머지는 금지한다(금기).** 이 스킬은 어떤 경우에도 `$DOBBY_DEFAULT_BASE`(정식 배포 베이스)로 PR을 열거나 머지하지 않는다. 사용자가 명시적으로 요청해도 이 스킬 범위에서는 수행하지 않고, 별도 절차로 사용자가 직접 진행하도록 안내만 한다. 통합은 상위 브랜치(`{prefix}/{상위키}`)까지만.

## 실행 모드

메인 에이전트 브랜치까지는 어느 모드든 커밋·푸시가 승인 없이 진행된다.

### 모드 A — 대화형 (Agent 도구 + SendMessage) · 기본
- 서브·리뷰 에이전트를 `Agent` 도구(`general-purpose`)로 기동. 워크트리 안에서 `CLAUDE.md` 등을 스스로 파악한다.
- 피드백 루프는 `SendMessage`로 **같은 에이전트를 이어가** 구현 컨텍스트를 유지한다.
- 단계 사이에 메인이 개입·검토하기 쉽다. 병렬 구현은 여러 `Agent` 호출을 한 메시지에 담아 동시에 띄운다.

### 모드 B — 자율 (Workflow)
- P4~P6(구현→리뷰→피드백 루프)을 `Workflow` 스크립트로 자동·병렬 실행한다. 각 하위이슈를 파이프라인 아이템으로, 리뷰·피드백을 스테이지로 구성한다.
- 손이 덜 가지만 중간 개입이 어렵다. 커밋·푸시가 메인 브랜치까지는 승인 없이 자동 진행된다(정식 베이스 반영은 여전히 승인 필요).
- Workflow는 명시적 opt-in이 필요하므로, `mode=B` 선택 자체를 opt-in으로 본다.

## 진행 모니터링

- **Phase 전이마다 `orchestration.md`를 갱신한다**: 에이전트 상태 표의 상태·라운드·`갱신 일시`를 바꾸고, `## 이벤트 로그`에 타임스탬프 1줄을 덧붙인다(P0 셋업, P1 분석 완료, P4 구현중 진입, P5 리뷰 완료·blocking 수, P6 라운드 종료, P7 통합 등).
- **상태 값은 `agent-start`의 "에이전트 상태 값(정식)"을 따른다**: 서브 에이전트 `대기→분석중→구현중→리뷰중→수정중→재통합대기→완료`, 리뷰 에이전트 `대기→진행중→완료`. 피드백 루프(P6)는 `리뷰중`↔`수정중`을 오가며 `라운드`를 올린다.
- **에이전트 작업 로그 위치 기록(부모가 수집)**: 서브·리뷰 에이전트(모드 A의 `Agent` 도구로 기동한 sidechain)의 대화 로그는 부모 세션 폴더 아래에 남는다:
  `~/.claude/projects/*/{부모세션ID}/subagents/agent-{agentId}.jsonl` (부모세션ID = `$CLAUDE_CODE_SESSION_ID`).
  실행 중인 서브 에이전트는 자기 `agentId`를 알 방법이 없으므로 **메인(부모)이 찾아서 기록**한다. 각 로그 파일의 첫 지시(역할 "어드민/지면/코드리뷰" + 이슈키)로 주인을 식별해, 슬러그→로그경로를 `$DOBBY_WORKSPACE/meta/.agent-start/{상위키}/agent-logs.json`에 남긴다. P4 기동 직후와 이후 갱신 시 채운다.
  ```bash
  SID="$CLAUDE_CODE_SESSION_ID"
  ls ~/.claude/projects/*/"$SID"/subagents/agent-*.jsonl   # 후보 로그들
  # 각 파일 첫 user 메시지의 역할·이슈키로 슬러그 매칭 → JSON 기록
  ```
  형식: `{ "{슬러그}": "{로그 절대경로}", ... }` (예: `{"admin-agent":"/Users/.../subagents/agent-a0400….jsonl"}`). 대시보드는 이 파일을 읽어 각 에이전트의 실제 작업 내역(수정 파일·커밋·요약)을 보여준다. **직접 변경 기록을 쓰지 않는다**(로그가 원본).
- 이렇게 하면 `갱신 일시`로 멈춤 여부를, `이벤트 로그`로 궤적을 파악할 수 있다.
- **상태 확인 한 줄**(관제탑 + 이슈별 상태 동시 확인):
  ```bash
  cat $DOBBY_WORKSPACE/meta/.agent-start/{상위키}/orchestration.md; \
  grep -H '상태' $DOBBY_WORKSPACE/meta/.issue-start/*/status.md
  ```
- 모드 B(자율)는 위 메타에 더해 `/workflows` 실시간 뷰로도 진행을 관측한다.

## 승인 게이트

- **승인 불필요(자동 진행)**: 하위이슈 브랜치와 **메인 에이전트 브랜치(`{prefix}/{상위키}`)** 까지의 커밋·푸시·머지, 그리고 그 브랜치들의 origin 푸시.
- **⛔ 금지(금기)**: `$DOBBY_DEFAULT_BASE`(정식 배포 베이스)로의 **PR 생성·머지**. 이 스킬에서는 절대 하지 않는다.
- **승인 필요**: 메인 브랜치 **밖으로 나가는 그 외 반영** — 배포(CD), 외부 서비스 반영 등. (정식 배포 베이스 PR/머지는 승인 대상이 아니라 아예 금지)

## 자율성 원칙

- 파일 오너십이 겹치지 않는 병렬 착수, 사실 확인(백그라운드 에이전트), 메인 브랜치까지의 커밋·푸시·머지는 **묻지 않고 진행**한다.
- 불확실한 사실은 사용자에게 묻기 전에 **백그라운드 에이전트로 먼저 확인**한다. 확인으로 판단 가능하면 그대로 진행한다.
- **자율 판단은 확인된 사실로만 한다. 추측으로 진행하지 않는다.** 백그라운드 확인 결과가 명확하지 않으면 다시 확인하고, 그래도 사실이 안 서면 그 지점을 사용자에게 알린다(추측으로 메우지 않는다).
- 사용자에게 확인이 필요한 경우는 좁다: 프론트 밖 작업(백엔드·디자인) 필요, 최대 라운드 초과, 확인으로도 결론이 안 서는 설계 분기. (`$DOBBY_DEFAULT_BASE` PR·머지는 확인 대상이 아니라 **금지**)

## 종료 조건

- 모든 하위이슈가 리뷰 클린 → 상위 브랜치 통합·푸시 완료 시 종료.
- 최대 라운드 초과, 프론트 밖 작업 필요, 확인으로도 안 풀리는 설계 분기는 사용자에게 알린다.

## 주의

- **사실 기반 원칙(최우선)**: 모든 판단·분석·범위 배분은 직접 확인한 사실(코드·문서·실행/조회 결과)에만 근거한다. **추측하지 않는다.** 확인 못 한 것은 "미확인"으로 명시하고, 필요하면 백그라운드 에이전트로 확인한다. 서브·리뷰 에이전트에게도 이 원칙을 지시로 전달한다.
- 어려운 용어를 지양하고 쉬운 말로 설명한다.
- 충돌 방지의 층위: 워크트리는 런타임 파일 경합을, 범위 화이트리스트는 **통합 시 머지 충돌**을 막는다.
- 계약(불변) / `status.md`(진행) / `orchestration.md`(관제탑)의 역할을 섞지 않는다.
- 서브 에이전트는 자기 브랜치만, 메인 에이전트는 상위 브랜치까지만 다룬다. 정식 베이스는 사용자 승인 후.
- 워크트리에는 `node_modules`가 없어 커밋 시 `--no-verify`로 훅을 건너뛴다.
