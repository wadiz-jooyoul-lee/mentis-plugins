# agent-impl

멀티 에이전트 오케스트레이션(→ `agent-start` 참조)에서 **서브 에이전트의 구현 단계**를 담당하는 스킬. 자신의 워크트리·브랜치에서 계약(contract)이 정한 범위 안에서만 구현하고, 진행 상태를 `status.md`에 남기며, 끝나면 커밋·푸시 후 메인 에이전트에게 리뷰를 요청하고 대기한다.

이 스킬을 실행하는 주체는 **서브 에이전트**다. 메인 에이전트가 기동 시 **계약 파일 경로**(`$DOBBY_META/.agent-start/{에픽키}/agents/{슬러그}.md`)를 함께 전달한다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 `~/.config/work-dobby/config.env`를 읽어 환경 변수를 불러온다(`[ -f ~/.config/work-dobby/config.env ] && source ~/.config/work-dobby/config.env`). **이미 값이 있는 변수는 묻지 않는다.** 빠진 변수만 아래 규칙으로 채운다.

- **기본값이 있는 변수**: 현재 기본값을 보여주고 그대로 쓸지 바꿀지 물어본다.
- **선택 변수**(생략 가능): 생략 시 어떤 영향이 있는지 설명하고 생략을 허용한다.
- 사용자가 정한 값은 설정 파일에 저장하고(`mkdir -p ~/.config/work-dobby` 후 기록) `export` 한다.

- **메타 루트**: `DOBBY_META="${DOBBY_META_PATH:-$DOBBY_WORKSPACE/meta}"` — `DOBBY_META_PATH`가 설정돼 있으면 그 경로, 없으면 `$DOBBY_WORKSPACE/meta`. 이하 메타 경로는 `$DOBBY_META` 기준.

| 변수 | 뜻 | 기본값 |
|------|-----|--------|
| `JIRA_BASE_URL` | Jira 사이트 주소 | `https://wadiz.atlassian.net` |
| `DOBBY_WORKSPACE` | 작업 루트(하위에 `subtree/`·`meta/`) | `$HOME/work/dobby-workspace` |
| `DOBBY_META_PATH` | 메타 경로 직접 지정(선택) | (없음 → `$DOBBY_WORKSPACE/meta`) |
| `DOBBY_DEFAULT_BASE` | 기본 베이스 브랜치 | `master` |
| `DOBBY_REPOS_ROOT` | 원본 소스 저장소들이 있는 루트 | `$HOME/work` |
| `DOBBY_ENV_MAP` | 테스트 환경→호스트 매핑 | `dev=dev.wadiz.io,rc=rc.wadiz.kr,rc2=rc2.wadiz.kr,rc4=rc4.wadiz.io` |
| `DOBBY_DOCS_ROOT` | 참고 문서 루트(선택) | (없음 → 참고 문서 없이 진행) |
| `TEST_LOGIN_ID` / `TEST_LOGIN_PW` | 테스트 계정(선택) | (없음 → 로그인 필요 테스트는 건너뜀) |

워크트리(작업 폴더)는 `$DOBBY_WORKSPACE/subtree/`, 메타 파일은 `$DOBBY_META/`에 둔다.

## 입력

- 계약 파일 경로(`agents/{슬러그}.md`). 전달받지 못했으면 메인 에이전트에게 요청한다.
- (선택) 리뷰 라운드 번호. 피드백 반영 실행이면 `reviews/round-{n}/{슬러그}.md`를 함께 읽는다.

## 절차

### 1. 컨텍스트 숙지 (구현 전)

- **계약 파일**(`agents/{슬러그}.md`)을 가장 먼저 읽는다. 특히 **수정 허용 범위(허용/금지/경계)**, 브랜치·워크트리 경로, 커밋 규칙, DoD를 확인한다.
- 담당 이슈의 `status.md`(`$DOBBY_META/.issue-start/{이슈키}/status.md`)의 **분석 상세·수정 설계**를 읽는다.
- 워크트리 루트의 `CLAUDE.md`(및 작업 디렉터리별 `CLAUDE.md`)를 읽어 코드·커밋 규약을 확인한다.
- 이슈 설명(Jira)이 필요하면 다시 조회한다.
- 분석 내용이 모호하거나 계약과 충돌하면 **임의로 판단하지 말고 메인 에이전트에게 확인**한다.

### 2. status.md 상태 전환 + 구현 요약

구현을 시작하기 전에 `status.md`를 갱신한다.

- 상태 값을 **`구현중`**으로 바꾸고 `갱신` 일시를 기록한다.
- **`## 구현` 섹션**에 아래 **요약 스냅샷**을 남긴다(이미 있으면 이어 쓴다). 파일별 상세한 변경 내역을 이 파일에 길게 적을 필요는 없다 — 실제 작업 로그는 메인 에이전트가 별도로 수집한다(→ `work-dobby`).

```markdown
## 구현
- **시작/갱신**: {YYYY-MM-DD HH:MM}
- **수정 범위(계약 대비)**: 허용 범위 안에서 실제로 건드린 파일 목록(경로만)
- **핵심 로직/설계 결정**: 구현하며 택한 방식과 이유(대안이 있었다면 함께)
- **경계/공용 파일 처리**: 경계 파일을 건드렸다면 무엇을·왜, 메인에 보고했는지
- **미해결/후속**: TODO, 확인이 필요한 부분
- **커밋**: {커밋 해시·메시지} / 푸시: {브랜치}
```

### 3. 구현

- **계약의 허용 범위 안에서만** 코드를 수정한다. 범위 밖 파일이 필요하면 임의로 건드리지 말고 메인 에이전트에게 요청한다(충돌 방지의 핵심).
- 기존 코드 패턴·`CLAUDE.md` 컨벤션을 따른다. 요청되지 않은 리팩터링·기능 추가는 하지 않는다(정밀한 수정).
- 구현하며 `## 구현` 요약을 갱신한다. **파일별 상세 변경 기록을 직접 작성하지 않는다** — 실제 수정 파일·커밋 등 구체 내역은 이 에이전트의 대화 로그에 그대로 남으며, 메인 에이전트가 그 로그 위치를 수집한다(→ `work-dobby`). 즉 여기서는 요약만 남기면 된다.

### 4. 커밋 + 푸시

- 워크트리에는 `node_modules`가 없어 pre-commit 훅이 실패할 수 있으므로 **`git commit --no-verify`**로 커밋한다.
- 커밋 메시지는 계약·프로젝트 규칙을 따른다(버그 `fix: {이슈키} ...`, 그 외 `feat: {이슈키} ...`). 커밋 메시지 스킬이 있으면 그 규약을 따른다.
- 자기 브랜치(`{prefix}/{이슈키}`)에 푸시한다. **상위(에픽) 브랜치에는 직접 푸시·머지하지 않는다**(머지는 메인 에이전트가 리뷰 통과 후 수행).
- 커밋 후 `## 구현` 요약에 커밋 해시·푸시 브랜치를 기록한다.

### 5. 리뷰 요청 + 대기

- 메인 에이전트에게 **리뷰를 요청하고 대기**한다. 요청에는 브랜치명, 요약(무엇을 구현했는지), `status.md` 경로를 포함한다.
- 메인 에이전트의 지시 없이 상위 브랜치를 건드리거나 다른 에이전트 범위를 수정하지 않는다.

### 6. 피드백 반영 (리뷰 라운드)

- `reviews/round-{n}/{슬러그}.md`를 읽어 지적 사항을 확인한다.
- **허용 범위 안에서** 수정하고, `## 구현` 요약을 이번 라운드 변경까지 갱신한다.
- 다시 `commit --no-verify` → 푸시 → 메인에 재리뷰 요청 후 대기.
- 범위 밖 수정이나 계약과 상충하는 지적은 메인 에이전트에게 확인 후 처리한다.

## 주의

- 이 스킬은 **자기 워크트리·브랜치 안에서만** 동작한다. 다른 에이전트의 워크트리·브랜치를 건드리지 않는다.
- 커밋·푸시는 이 흐름(오케스트레이션)에서 허용된 동작이다. 단 **상위 브랜치 머지·푸시는 하지 않는다**.
- 계약의 허용 범위를 벗어나는 판단이 필요하면 반드시 메인 에이전트에게 확인한다.
- **사실 기반 원칙(최우선)**: 구현·판단은 직접 확인한 사실(코드·문서·실행 결과)에만 근거한다. **추측하지 않는다.** 확인 못 한 동작은 코드를 읽어 확인하고, 확인 불가하면 `## 구현`의 `미해결/후속`에 "미확인"으로 남긴다. 애매하면 메인 에이전트에게 확인한다.
- 어려운 용어를 지양한다.
- `status.md`의 `## 구현`은 **요약 스냅샷**으로 유지한다. 파일별 구체 변경 내역은 이 에이전트의 대화 로그에 그대로 남으므로, `status.md`에 장황하게 옮겨 적지 않는다(요약이면 충분).
