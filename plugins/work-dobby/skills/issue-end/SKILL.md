# issue-end

`/issue-start`로 만든 이슈 워크트리(`$DOBBY_WORKSPACE/subtree/` 안의 폴더들)를 정리하는 스킬. **종료된 Jira 이슈**의 워크트리만 찾아 안전하게 제거한다. 진행 중인 이슈와 미푸시 커밋이 남은 폴더는 건드리지 않는다.

> 워크트리(worktree): 하나의 git 저장소에서 여러 작업 폴더를 동시에 두는 기능. `git worktree remove`로 제거해도 **브랜치는 그대로 보존**되며, 필요하면 같은 브랜치로 워크트리를 다시 만들 수 있다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 **`${CLAUDE_PLUGIN_ROOT}/reference/config.md`의 "설정 절차"를 그대로 따른다**: `~/.config/work-dobby/config.env`를 source 해 환경 변수를 불러오고, **이미 값이 있는 변수는 묻지 않고** 빠진 값만 규칙대로 채워 저장·export 한다. 메타 루트 `DOBBY_META`, 변수 목록·기본값, 폴더 배치(워크트리 `$DOBBY_WORKSPACE/subtree/` · 메타 `$DOBBY_META/`)는 모두 그 문서에 있다. 이하 메타 경로는 `$DOBBY_META` 기준.

## 입력

- `args`(선택): 이슈 키(`QA-22370`). 주면 **그 이슈 폴더만** 대상으로 한다.
- 없으면 `$DOBBY_WORKSPACE/subtree/` 안의 **모든** 이슈 워크트리를 대상으로 한다.

## 처리 범위 (안전 경계)

- **오직 `$DOBBY_WORKSPACE/subtree/` 하위 폴더만** 다룬다. 원본 소스 repo(`$DOBBY_REPOS_ROOT/{repo}` 등)나 그 밖의 폴더는 절대 건드리지 않는다.
- 종료(완료) 상태가 **아닌** 이슈, 그리고 **미푸시 커밋이 남은** 폴더는 삭제하지 않는다. 미커밋(작업트리) 변경은 종료 이슈면 삭제를 막지 않는다(제거 시 버려짐).

## 절차

### 1. subtree 폴더 스캔

- `$DOBBY_WORKSPACE/subtree/`가 없으면 "정리할 워크트리가 없다"고 알리고 종료한다.
- 하위 폴더 목록을 읽는다. 각 폴더에 대해:
  - 폴더명에서 **이슈키**를 추출한다. 폴더명 규칙은 `{repo}-{이슈키}`이며 이슈키는 `대문자+숫자` 형태다(정규식 예: `[A-Z]+-[0-9]+` 의 마지막 매치). 예: `{repo}-QA-22370` → `QA-22370`, `{repo2}-QA-22413` → `QA-22413`.
  - 이슈키를 못 뽑는 폴더는 건너뛰고 목록에 "판단 불가"로 보고한다(자동 삭제 금지).
  - `args`가 주어졌으면 그 이슈키와 일치하는 폴더만 남긴다.

### 2. 워크트리 상태 확인 (폴더별)

각 대상 폴더에 대해 사실을 직접 확인한다(추측 금지):

- 워크트리인지 확인: `.git` 파일이 `gitdir: .../worktrees/...`를 가리키는지. 소속 원본 repo 경로도 함께 파악.
- 현재 브랜치: `git -C {폴더} rev-parse --abbrev-ref HEAD`
- **미커밋 변경**: `git -C {폴더} status --porcelain` (출력이 있으면 변경 있음)
- **미푸시 커밋**: `git -C {폴더} log --oneline @{u}..` (업스트림 대비 안 올라간 커밋). 업스트림이 없으면 그 사실을 기록.

### 3. Jira 상태 일괄 조회

- 추출한 이슈키들의 현재 상태를 조회한다. 특정 에픽에 묶지 말고 **각 이슈 자기 상태**를 본다(QA·FE1 등 프로젝트 무관).
- JQL: `key in (KEY1, KEY2, ...)`. 응답이 크면 파일로 저장될 수 있으니 `jq`로 필요한 필드만 뽑는다:
  - `mcp__claude_ai_Atlassian__searchJiraIssuesUsingJql` (cloudId: `$JIRA_BASE_URL`의 호스트, fields: `["key","status"]`)
  - 종료 판정 기준: **`status.statusCategory.key == "done"`** (상태명 "종료" 등). "진행 중"(`indeterminate`)·"할 일"(`new`)은 유지.
- 폴더는 있는데 Jira에서 이슈를 못 찾으면 "조회 불가"로 보고하고 자동 삭제하지 않는다.

### 4. 삭제 대상 선별 + 보고

- 삭제 대상 = (이슈가 **종료**) AND (미푸시 커밋 없음). 미커밋(작업트리) 변경 여부는 삭제 판정에 넣지 않는다.
- 처리 전에 **표로 요약**해 보여준다: 폴더 / 이슈 상태 / dirty·unpushed 여부 / 처리(삭제·유지). dirty(미커밋)는 참고용으로 표시하되 삭제를 막지는 않는다.
  - 종료지만 **미푸시 커밋**이 있는 폴더는 **유지**하고 사유를 명확히 표시한다(사용자가 따로 정리하도록).
  - 진행 중 이슈는 유지.

### 5. 코드 변경 스냅샷 저장 (제거 전, 필수)

워크트리를 제거하면 코드 변경을 더는 볼 수 없으므로, **제거하기 전에** 각 워크트리의 변경 내용을 메타에 스냅샷으로 남긴다(이후 대시보드 등에서 워크트리가 사라져도 조회 가능).

- 저장 위치: `$DOBBY_META/.issue-end/{이슈키}/code-changes/` (없으면 `mkdir -p`)
- 베이스 `{base}`는 `.issue-start/{이슈키}/status.md`의 `베이스` 값을 쓴다(없으면 `$DOBBY_DEFAULT_BASE`). **`origin/{base}`를 우선** 사용하고(로컬 base는 오래됐을 수 있음), 없으면 로컬 `{base}`를 쓴다.
- 워크트리(repo)마다 아래를 실행해 저장한다(제거 대상뿐 아니라, 종료 처리하는 이슈의 워크트리 전부):
  ```bash
  git -C {워크트리} log --oneline {base}..HEAD  > $DOBBY_META/.issue-end/{이슈키}/code-changes/{repo}.commits
  git -C {워크트리} diff {base}...HEAD          > $DOBBY_META/.issue-end/{이슈키}/code-changes/{repo}.diff
  ```
- 변경/커밋이 없으면 빈 파일이 남아도 무방하다. 실패(ref 없음 등)하면 사유를 서머리에 한 줄로 남기고 계속 진행한다.

### 6. 제거 실행

- 삭제 대상 각 폴더를 소속 원본 repo에서 정식 제거한다: `git -C {원본repo} worktree remove {폴더 절대경로}`
  - 이렇게 하면 폴더 삭제 + 워크트리 등록정보 정리가 함께 되고, **브랜치는 보존**된다.
  - 미커밋(작업트리) 변경이 남아 `worktree remove`가 거부되면, 종료 이슈이므로 그 변경은 버리기로 한 것이니 `--force`로 제거한다(미푸시 커밋이 있는 폴더는 4단계에서 이미 유지 처리되어 여기 오지 않는다).
- 등록정보가 깨져 `worktree remove`가 실패하면, 사용자에게 보고 후 동의를 받아 `rm -rf` + 해당 repo에서 `git worktree prune`으로 정리한다(임의로 강제 삭제하지 않는다).

### 7. 종료 서머리 저장

- **종료(Jira `done`)로 판정된 이슈마다** 종료 서머리를 남긴다(워크트리 실제 제거 여부와 무관 — 종료됐지만 미커밋·미푸시로 유지된 경우도 서머리는 남기고 "유지" 사유를 기록).
  - 경로: `$DOBBY_META/.issue-end/{이슈키}/summary.md` (없으면 `mkdir -p`)
- **issue-start 메타 이어받기**: `$DOBBY_META/.issue-start/{이슈키}/status.md`가 있으면 읽어 이슈 메타(제목·타입)와 작업 요약(원인 위치·수정 설계)을 서머리로 옮긴다.
  - 없으면(예전 이슈 등) 이어받지 못했다고 표시하고, issue-end가 아는 정보(Jira 상태·워크트리)만 채운다.
- 종료 시점 정보(최종 상태명, 종료 처리 일시, 제거/유지 결과)를 덧붙인다. 규격은 아래 "종료 서머리 (summary.md)" 참조.
- `.issue-start/{이슈키}/`는 **삭제하지 않는다**(생명주기 원본 기록). 워크트리 폴더만 제거 대상이다.

### 8. 결과 정리

- 삭제한 폴더, 유지한 폴더(사유 포함), 저장한 종료 서머리 경로를 표로 요약한다.
- 남은 `$DOBBY_WORKSPACE/subtree/` 목록을 보여준다.

## 종료 서머리 (summary.md)

- 경로: `$DOBBY_META/.issue-end/{이슈키}/summary.md` (종료 이슈당 하나)
- 목적: 워크트리(코드 폴더)는 지워도 **"이 이슈로 무엇을 했고 어떻게 끝났는지"는 히스토리로 보존**. issue-start의 메타를 이어받아 종료 정보를 더한다.

```markdown
# {이슈키} 종료 서머리

## 이슈
- **키**: {이슈키} · **타입**: {버그|작업|…} · **제목**: {제목}
- **Jira**: $JIRA_BASE_URL/browse/{이슈키}
- **최종 상태**: {상태명} (statusCategory: done)

## 종료 처리
- **처리 일시**: {YYYY-MM-DD HH:MM}
- **워크트리 처리**: 제거 / 유지(사유: 미푸시 커밋 등)

| repo | 브랜치(보존) | 경로 | 처리 |
|------|-------------|------|------|
| {repo} | {prefix}/{이슈키} | $DOBBY_WORKSPACE/subtree/{repo}-{이슈키} | 제거 |

## 작업 요약 (issue-start에서 이어받음)
- **원인 위치**: `파일:라인`
- **수정 설계**: {요약}
- (issue-start 기록이 없으면: "issue-start 상태 파일 없음 — 메타 미상")
```

- 브랜치는 `git worktree remove` 후에도 보존되므로 표에 "보존"으로 표기한다.
- 종료됐지만 미푸시 커밋으로 워크트리를 유지한 경우, 처리를 "유지"로 적고 사유를 남긴다(서머리는 그대로 저장).

## 주의

- **`$DOBBY_WORKSPACE/subtree/` 밖은 절대 건드리지 않는다.** 원본 소스 repo·기타 폴더 보호.
- **종료 서머리(`.issue-end/`)와 생명주기 기록(`.issue-start/`)은 삭제하지 않는다.** 워크트리(코드 폴더)만 제거 대상이다.
- 미푸시 커밋이 남은 폴더는 종료 이슈라도 자동 삭제하지 않고 보고만 한다. 미커밋(작업트리) 변경은 종료 이슈면 삭제 시 버려진다(제거 전 5단계 스냅샷으로 커밋된 diff는 보존됨).
- `git worktree remove`는 브랜치를 지우지 않는다. 브랜치까지 정리하려면 사용자에게 따로 확인받는다.
- 종료 판정은 상태명이 아니라 `statusCategory.key == "done"`으로 한다(상태명이 바뀌어도 안전).
- 어려운 용어는 쉬운 말로 풀어 설명한다.
