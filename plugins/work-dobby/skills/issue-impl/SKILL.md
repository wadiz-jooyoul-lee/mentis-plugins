# issue-impl

`issue-start`가 도출한 **분석·수정 설계를 실제 코드로 구현**하고, 완성 코드를 **적대적 코드리뷰**로 스스로 걸러 커밋·푸시하고, 리뷰가 안정되면(blocker·major 0) 사용자 동의 후 **베이스 브랜치로 머지**까지 하는 스킬. 오케스트레이션(멀티 에이전트) 없이 **단독 이슈 한 건**을 처리한다.

> 오케스트레이션으로 여러 에이전트가 병렬 구현하는 경우는 `agent-start`/`i-order-you-to-develop`·`agent-impl`을 쓴다. `issue-impl`은 그 흐름과 무관한 **단독 구현 경로**다.

생명주기 안에서의 위치:

```
issue-start(분석·수정 설계)  →  [issue-impl: 구현 + 셀프 코드리뷰 + 반영 + 베이스 머지]  →  issue-test(검증)  →  issue-end(정리)
```

**핵심 목적:** PR을 올린 뒤 CI 코드리뷰 봇이 지적하는 수정사항이 잦다. 그 봇과 **같은 기준**으로 PR 전에 미리 리뷰·수정해 리뷰 churn을 줄인다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 `~/.config/work-dobby/config.env`를 읽어 환경 변수를 불러온다(`[ -f ~/.config/work-dobby/config.env ] && source ~/.config/work-dobby/config.env`). **이미 값이 있는 변수는 묻지 않는다.** 빠진 변수만 아래 규칙으로 채운다.

- **기본값이 있는 변수**: 현재 기본값을 보여주고 그대로 쓸지 바꿀지 물어본다.
- **선택 변수**(생략 가능): 생략 시 어떤 영향이 있는지 설명하고 생략을 허용한다.
- 사용자가 정한 값은 설정 파일에 저장하고(`mkdir -p ~/.config/work-dobby` 후 기록) `export` 한다.

- **메타 루트**: `DOBBY_META="${DOBBY_META_PATH:-$DOBBY_WORKSPACE/meta}"` — `DOBBY_META_PATH`가 설정돼 있으면 그 경로, 없으면 `$DOBBY_WORKSPACE/meta`. 이하 메타 경로는 `$DOBBY_META` 기준.

| 변수 | 뜻 | 기본값 |
|------|-----|--------|
| `JIRA_BASE_URL` | Jira 사이트 주소 | `https://wadiz.atlassian.net` |
| `DOBBY_WORKSPACE` | 작업 루트(하위에 `subtree/`·`meta/`) | `$HOME/work/dobby-workspace` |
| `DOBBY_META_PATH` | 메타 경로 직접 지정(선택) | (없음 → `$DOBBY_WORKSPACE/meta`) |
| `DOBBY_DEFAULT_BASE` | 기본 베이스 브랜치 | `master` |
| `DOBBY_REPOS_ROOT` | 원본 소스 저장소들이 있는 루트 | `$HOME/work` |
| `DOBBY_DOCS_ROOT` | 참고 문서 루트(선택) | (없음 → 참고 문서 없이 진행) |

워크트리(작업 폴더)는 `$DOBBY_WORKSPACE/subtree/`, 메타 파일은 `$DOBBY_META/`에 둔다.

## 입력

- `args`: 이슈 번호(`QA-22370`) 또는 Jira URL. 없으면 물어본다.
- URL이면 `/browse/{KEY}`에서 이슈 키를 추출한다.

## 사전 조건

- **`issue-start`의 상태 파일이 있어야 한다**: `$DOBBY_META/.issue-start/{이슈키}/status.md`.
  - 없으면 "먼저 `/issue-start {이슈키}`로 착수·분석하라"고 안내하고 중단한다(임의로 분석부터 시작하지 않는다).
  - 상태가 `착수`(분석 미완료)면 분석이 얕을 수 있으니 사용자에게 그대로 진행할지 확인한다.
- status.md의 **워크트리 경로**가 실제로 존재해야 한다. 없으면(정리됨) 사용자에게 알리고 재생성 여부를 확인한다.

## 절차

### 1. 컨텍스트 숙지 (구현 전)

- **`status.md`의 `## 분석 상세`(원인·수정 설계)를 가장 먼저 읽는다.** 이 스킬은 여기 적힌 설계를 구현하는 것이 목적이다.
- status.md의 **워크트리/브랜치 표**에서 대상 워크트리 경로·브랜치·베이스를 확인한다. 이후 모든 작업은 그 워크트리 안에서 한다.
- **리뷰·코딩 기준 문서를 수집한다**(3단계 리뷰에서 그대로 쓴다). 워크트리 기준으로:
  - 루트 및 대상 파일이 속한 각 영역의 `CLAUDE.md`(예: `apps/CLAUDE.md`, `apps/{앱}/CLAUDE.md`, `packages/CLAUDE.md`)
  - `NAMING_CONVENTIONS.md`, `GLOSSARY.md`
  - `.github/workflows/`의 코드리뷰 워크플로우가 있으면 그 **프로젝트 고유 규칙**(예: i18n 특례)을 읽어 반영한다.
- 설계가 모호하거나 실제 코드와 어긋나면 **임의로 판단하지 말고** 사용자에게 확인한다. 확인 없이 설계를 벗어난 구현을 하지 않는다.

### 2. 구현 (status.md → 구현중)

- status.md 상태를 **`구현중`**으로 바꾸고 `갱신` 일시를 기록한다.
- **수정 설계대로** 구현한다. 요청되지 않은 리팩터링·기능 추가는 하지 않는다(정밀한 수정). 기존 코드 패턴과 영역 `CLAUDE.md` 규약을 따른다.
- 구현하며 status.md의 `## 구현` 섹션에 **요약 스냅샷**(건드린 파일·설계 결정·미해결)을 남긴다. 파일별 장황한 diff를 옮겨 적지 않는다.

### 3. 셀프 코드리뷰 (별도 리뷰 에이전트, 적대적)

구현 직후, **자기 세션이 아니라 별도 서브에이전트**를 띄워 리뷰한다(자기 코드 편향 방지). Agent(Task) 도구로 리뷰 에이전트를 기동하고, 아래를 전달한다.

- **리뷰 대상 diff**: 워크트리에서 `git diff {fork}..HEAD` + 커밋 안 한 변경(`git diff HEAD`)까지. (`fork`는 브랜치가 base에서 갈라진 지점 — `git merge-base origin/{base} HEAD`)
- **컨텍스트**: status.md의 원인·수정 설계, 1단계에서 수집한 CLAUDE.md/NAMING/GLOSSARY·CI 고유 규칙 경로.
- **지시**: 아래 "리뷰 루브릭"으로 diff를 **적대적으로(문제를 찾아내는 관점에서)** 검토하고, 지적을 **심각도·근거·수정안**과 함께 목록으로 낼 것. 추측이면 추측이라 표시할 것.

리뷰 에이전트는 구현하지 않고 **findings 목록만** 반환한다.

### 4. 지적 반영 (안전 자동 / 애매 확인)

리뷰 findings를 심각도로 분류해 반영한다.

- **자동 반영**: 규약 위반·명백한 오류·nit 등 **해석의 여지가 없는 것**은 바로 수정한다.
- **사용자 확인 후 반영**: 설계 변경이 필요하거나, 해석이 갈리거나, 수정 설계와 상충하는 지적은 **임의로 고치지 말고** 사용자에게 요약해 확인받는다.
- 반영 결과를 status.md `## 셀프리뷰` 섹션에 기록한다(아래 규격).
- 반영으로 코드가 바뀌었으면 **3단계 리뷰를 한 번 더** 돌려(달라진 부분 위주) blocker/major가 남지 않았는지 확인한다. 남으면 다시 반영 → 재리뷰. **blocker·major가 0이 될 때까지** 반복하되, 무한 반복을 막기 위해 라운드는 최대 3회로 하고 그래도 남으면 사용자에게 보고한다.

### 5. 검증 + 커밋 + 푸시

- 영역 `CLAUDE.md`에 적힌 **검증 명령**(lint/type-check/build/test)이 있으면 그 워크트리에서 실행해 통과를 확인한다. 실패하면 고치고, 못 고치면 status.md에 남기고 사용자에게 보고한다.
- 워크트리에는 `node_modules`가 없어 pre-commit 훅이 실패할 수 있으므로 **`git commit --no-verify`**로 커밋한다. (검증 명령 실행에 의존성이 필요하면 `yarn install` 안내만 하고 자동 실행하지 않는다.)
- 커밋 메시지는 `commit-message` 스킬 규약을 따른다(버그 `fix: {이슈키} ...`, 그 외 `feat: {이슈키} ...`).
- 먼저 **자기 브랜치(`{prefix}/{이슈키}`)에 푸시한다.** **PR 생성은 하지 않는다**(사용자가 별도로 수행).
- 커밋·푸시는 이 흐름에서 허용된 동작이나, 규약상 실제 실행 직전에 대상 브랜치·커밋 메시지를 사용자에게 한 번 보여준다.

### 6. 베이스 브랜치 머지 (리뷰 안정 시, 사용자 동의 필수)

셀프 코드리뷰가 **안정적으로 끝난 경우에만**(4단계 최종 라운드에서 **blocker·major 0**) 자기 브랜치를 베이스 브랜치로 머지한다. blocker/major가 남아 있으면 머지하지 않고 중단한다.

- **동의 게이트**: 머지는 공유 브랜치(베이스)를 바꾸므로, 실행 직전 **대상 베이스 브랜치·머지할 이슈 브랜치·머지 방식을 사용자에게 보여주고 동의를 받는다.** 동의 없이 머지하지 않는다.
- **베이스 확인**: status.md 워크트리/브랜치 표의 **베이스**를 대상으로 한다(예: `master` 또는 `feature/{에픽}`).
- **최신화·안전 점검**: `git fetch origin {base}` 후, 베이스 체크아웃(베이스 워크트리가 있으면 그곳, 없으면 사용자에게 확인)이 **클린**한지 확인한다. 커밋 안 된 변경이 있으면 건드리지 말고 사용자에게 알린다.
- **머지 실행**: 베이스에서 `git merge --no-ff {prefix}/{이슈키}`로 머지(히스토리 보존)하고 `git push origin {base}`. **충돌이 나면 강제로 풀지 말고 중단**해 사용자에게 보고한다.
- **금지**: 상위 통합 브랜치를 이슈 브랜치에 역머지하지 않는다. PR 생성도 하지 않는다.

### 7. 마무리 기록

- status.md `## 구현`에 최종 커밋 해시·푸시 브랜치, **베이스 머지 결과(머지 커밋·푸시 브랜치)** 를 기록한다.
- `## 셀프리뷰`에 리뷰 라운드·주요 findings·반영 결과를 남긴다.
- 상태는 구현·리뷰·머지가 끝나도 **`구현중` 유지**한다. `해결됨`은 `/issue-test`가 검증을 통과시킬 때만 바꾼다(생명주기 규칙 유지). 다음 단계로 `/issue-test {이슈키}`를 안내한다.

## 리뷰 루브릭 (내장 — 3단계 리뷰 에이전트가 그대로 사용)

CI 코드리뷰 봇이 지적하는 내용을 미리 잡는 것이 목표다. 아래 축으로 검토한다.

### A. 정확성 / 품질
- 버그·논리 오류, `null`/`undefined`·경계값 미처리, 비동기/에러 처리 누락.
- 회귀 위험: 같은 코드를 쓰는 다른 곳(소비처)에 미치는 영향.
- 불필요한 복잡성·중복, 이 변경으로 생긴 데드 코드(임포트·변수·함수).
- 수정 설계와 실제 구현의 불일치.

### B. 규약 적합성 (프로젝트 CLAUDE.md / NAMING / GLOSSARY)
- 명시적 요청 없이 기존 코드 패턴을 바꾸지 않았는가.
- 조건문은 긍정문 우선, 단일 if에도 중괄호 `{}`.
- 타입 import는 별도 행 `import type`. React 내장 타입은 전역 `React` 네임스페이스로 참조.
- 컴포넌트 작성 순서(선언 → 메서드 → 핸들러 → `useEffect` → 렌더), effect가 의존 함수보다 아래.
- 이미 보장된 조건에 중복 가드 추가 금지.
- 불필요한 파일 분리 지양(강결합 타입·함수는 한 파일).
- 주석은 필요한 코드에만, 한국어·존댓말로 간결하게.
- 네이밍은 `NAMING_CONVENTIONS.md`, 용어는 `GLOSSARY.md`를 따른다.
- **위 항목은 예시다. 실제로는 그 워크트리·영역의 CLAUDE.md/NAMING을 직접 읽어 그 프로젝트 규칙으로 대조한다.**

### C. CI 고유 규칙 (`.github/workflows`에서 읽어 반영)
- 예) `packages/i18n/**`: **값(번역문)은 리뷰하지 않는다.** 키가 **rename/삭제**된 경우만 지적한다. 추가된 키·값 변경·포맷은 지적하지 않는다.
- 프로젝트마다 다르므로 워크플로우의 실제 규칙을 확인해 그대로 적용한다.

### D. 커밋 메시지
- `commit-message` 스킬 규약 준수(`fix:`/`feat: {이슈키} ...`), 의미 단위 커밋.

### 심각도 분류
- **blocker**: 병합 불가(버그·회귀·설계 위반). 반드시 반영.
- **major**: 반영 강권(명백한 규약 위반·품질 문제).
- **minor / nit**: 사소·취향. 자동 반영 가능하면 반영, 아니면 기록만.

## status.md `## 셀프리뷰` 규격

```markdown
## 셀프리뷰
- **리뷰 일시/라운드**: {YYYY-MM-DD HH:MM} · round {n}
- **리뷰 기준**: 내장 루브릭 + {읽은 CLAUDE.md/NAMING 경로} + CI 규칙({있으면})
- **findings**
  | # | 심각도 | 파일:라인 | 지적 | 반영 |
  |---|--------|-----------|------|------|
  | 1 | blocker | a.tsx:12 | ... | 반영함(커밋 …) |
  | 2 | nit | b.ts:3 | ... | 기록만(사유) |
- **남은 이슈**: {blocker/major 0 여부, 사용자 확인 대기 항목}
```

## 주의

- **사실 기반 원칙(최우선)**: 구현·리뷰·판단은 직접 확인한 사실(코드·문서·실행 결과)에만 근거한다. **추측하지 않는다.** 애매하면 사용자에게 확인한다.
- 이 스킬은 **자기 이슈의 워크트리·브랜치 안에서만** 동작한다.
- 커밋·푸시는 허용되나 **PR 생성은 하지 않는다**. 커밋·푸시는 사용자 동의 후 진행한다. **베이스 브랜치 머지는 리뷰 안정(blocker·major 0) + 사용자 동의**를 모두 만족할 때만 수행한다(6단계).
- 리뷰는 **반드시 별도 서브에이전트**로 수행한다(자기 코드 편향 방지). 인라인으로 대체하지 않는다.
- 어려운 용어를 지양하고 쉬운 말로 설명한다. 전문용어는 처음 쓸 때 뜻을 덧붙인다.
- 구현 후 검증·테스트는 이 스킬의 lint/type/build 확인으로 끝이 아니다. 기능 검증은 `/issue-test`가 담당한다.
