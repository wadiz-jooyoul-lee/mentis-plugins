# agent-start

`issue-start`를 멀티 에이전트 오케스트레이션에 맞게 확장한 스킬. 하위이슈를 상위(에픽) 브랜치 위에 올려 서브 에이전트로 구현시키는 흐름에서, **각 하위이슈의 착수+분석**을 담당한다. 실제 코드 수정·커밋·푸시는 하지 않는다(구현은 `agent-impl`를 쓰는 서브 에이전트 몫).

> 워크트리(worktree): 하나의 git 저장소에서 여러 작업 폴더를 동시에 두는 기능. 이슈마다 별도 폴더를 만들어 메인 폴더를 건드리지 않고 작업한다.

`issue-start`와의 차이는 **① 커스텀 베이스 브랜치 지원**, **② 상위 에픽 단위 오케스트레이션 메타 규격 정의**뿐이다. 나머지 절차(Jira 전환·분석·테스트 목록)는 `issue-start`와 동일하다.

## 설정 (첫 실행 시 확인)

작업을 시작하기 전에 `~/.config/work-dobby/config.env`를 읽어 환경 변수를 불러온다(`[ -f ~/.config/work-dobby/config.env ] && source ~/.config/work-dobby/config.env`). **이미 값이 있는 변수는 묻지 않는다.** 빠진 변수만 아래 규칙으로 채운다.

- **기본값이 있는 변수**: 현재 기본값을 보여주고 그대로 쓸지 바꿀지 물어본다.
- **선택 변수**(생략 가능): 생략 시 어떤 영향이 있는지 설명하고 생략을 허용한다.
- 사용자가 정한 값은 설정 파일에 저장하고(`mkdir -p ~/.config/work-dobby` 후 기록) `export` 한다.

- **메타 루트**: `DOBBY_META="${DOBBY_META_PATH:-$DOBBY_WORKSPACE/meta}"` — `DOBBY_META_PATH`가 설정돼 있으면 그 경로, 없으면 `$DOBBY_WORKSPACE/meta`. 이하 메타 경로는 `$DOBBY_META` 기준.

| 변수 | 뜻 | 기본값 |
|------|-----|--------|
| `JIRA_BASE_URL` | Jira 사이트 주소 | `https://wadiz.atlassian.net` |
| `DOBBY_WORKSPACE` | 작업 루트(하위에 `subtree/`·`meta/`) | `$HOME/work/dobby-workspace` |
| `DOBBY_META_PATH` | 메타 경로 직접 지정(선택) | (없음 → `$DOBBY_WORKSPACE/meta`) |
| `DOBBY_DEFAULT_BASE` | 기본 베이스 브랜치 | `master` |
| `DOBBY_REPOS_ROOT` | 원본 소스 저장소들이 있는 루트 | `$HOME/work` |
| `DOBBY_ENV_MAP` | 테스트 환경→호스트 매핑 | `dev=dev.wadiz.io,rc=rc.wadiz.kr,rc2=rc2.wadiz.kr,rc4=rc4.wadiz.io` |
| `DOBBY_DOCS_ROOT` | 참고 문서 루트(선택) | (없음 → 참고 문서 없이 진행) |
| `TEST_LOGIN_ID` / `TEST_LOGIN_PW` | 테스트 계정(선택) | (없음 → 로그인 필요 테스트는 건너뜀) |

워크트리(작업 폴더)는 `$DOBBY_WORKSPACE/subtree/`, 메타 파일은 `$DOBBY_META/`에 둔다.

## 입력

- `args`: `{이슈키|URL} [base={브랜치}]`
  - 이슈 키(`FE1-1187`) 또는 Jira URL(`$JIRA_BASE_URL/browse/FE1-1187`). URL이면 `/browse/{KEY}`에서 키 추출. 없으면 물어본다.
  - `base={브랜치}`: 워크트리를 만들 베이스 브랜치. 예: `base=feature/FE1-1186`. **미지정 시** 기본 `$DOBBY_DEFAULT_BASE`(기본값 master).

## 절차

### 1. 이슈 조회 + 진행 중 전환

`issue-start` 1단계와 동일하다.
- `mcp__mcp-atlassian__jira_get_issue`로 제목·설명·타입·**현재 상태**를 조회(`expand: renderedFields`). 설명에 이미지가 있으면 `jira_get_issue_images`로 확인.
- 이슈 상태를 **`statusCategory.key`로 판정**해 "진행 중"으로 전환한다. `indeterminate`면 그대로, `done`이면 전환하지 말고 사용자에게 확인, `new`면 `jira_get_transitions`에서 도착 상태 `statusCategory.key == "indeterminate"`인 전이를 골라 `jira_transition_issue`로 전환. 애매하면 사용자에게 확인.

### 2. 베이스 브랜치 결정

- **`base=` 인자가 있으면 그 값을 그대로 `{base}`로 쓴다.**
  - 어느 베이스로 잡았는지 사용자에게 밝힌다.
- `base=` 인자가 없으면 기본 베이스 `$DOBBY_DEFAULT_BASE`(기본값 master)를 쓴다. 애매하면 임의로 정하지 말고 사용자에게 확인한다.
- `{base}`는 **로컬 브랜치**(예: `feature/FE1-1186`) 또는 **원격 참조**(예: `$DOBBY_DEFAULT_BASE` → `origin/$DOBBY_DEFAULT_BASE`)일 수 있다. 4단계에서 존재 여부로 갈라 처리한다.

### 3. 브랜치 prefix 결정 (이슈 타입)

- 버그 → `bugfix/{이슈키}`, 그 외(작업 등) → `feature/{이슈키}`.

### 4. 워크트리 + 이슈 브랜치 생성 (커스텀 베이스)

- **소스 저장소 루트 결정**: 소스 루트는 `$DOBBY_REPOS_ROOT/{repo}`(기본 `$HOME/work`)로 잡는다. 거기에 없으면 `git rev-parse --show-toplevel`로 확인한다. 이후 모든 git 명령은 소스 루트에서(`cd {sourceRoot}`) 실행한다.
- 워크트리는 `$DOBBY_WORKSPACE/subtree/{repo}-{이슈키}`에 만든다(없으면 `mkdir -p $DOBBY_WORKSPACE/subtree`).
- **중복 확인**: 소스 루트에서 `git worktree list`·`git branch --list {prefix}/{이슈키}`로 확인. 이미 있으면 `issue-start`의 "메타 backfill" 규칙을 그대로 적용한다(메타 있으면 사용자 확인 후 중단, 메타 없으면 기존 워크트리 재사용 + 분석/메타만 채움, fetch/worktree add는 건너뜀).
- **베이스 준비 및 워크트리 생성** — `{base}`의 형태로 갈라 처리:
  - `{base}`가 **로컬 브랜치로 존재**하면(예: 방금 만든 `feature/FE1-1186`): 원격 최신화가 필요하면 안내만 하고, 로컬 브랜치에서 바로 만든다.
    `git worktree add -b {prefix}/{이슈키} $DOBBY_WORKSPACE/subtree/{repo}-{이슈키} {base}`
  - `{base}`가 **원격 기준 브랜치**(`$DOBBY_DEFAULT_BASE` 등)면: `git fetch origin {base}` 후
    `git worktree add -b {prefix}/{이슈키} $DOBBY_WORKSPACE/subtree/{repo}-{이슈키} origin/{base}`
  - 로컬·원격 어디에도 `{base}`가 없으면 **중단하고 사용자에게 알린다**(베이스가 아직 안 만들어졌을 수 있음).
- **의존성 안내**: 새 워크트리는 `node_modules`가 비어 있을 수 있다. 필요 시 `yarn install`을 안내만 하고 자동 실행하지 않는다.
- 커밋·푸시는 하지 않는다.
- **status.md 초기화**: `$DOBBY_META/.issue-start/{이슈키}/status.md`(없으면 `mkdir -p`). 이슈 메타·상태 `착수`·Jira 전환 결과·워크트리/브랜치·**베이스**(`{base}` 값 그대로)·시작 일시를 기록한다. 규격은 아래 "status.md" 참조.

### 5. 분석 (코드 위치까지)

`issue-start` 5단계와 동일하다.
- 탐색·분석은 새 워크트리 폴더 기준. `DOBBY_DOCS_ROOT`가 설정돼 있으면 코드 분석 전에 `{DOBBY_DOCS_ROOT}/{repo}` 문서를 먼저 읽는다(미설정이면 참고 문서 없이 진행).
- 이슈 내용 기준으로 관련 코드·컴포넌트를 Grep/Glob으로 찾아 **`파일:라인`까지 특정**. 추측하지 않고 코드로 확인한다.

### 6. 결과 정리 + 수정 설계 제시

`issue-start` 6단계와 동일하다.
- 분석 결과(원인·`파일:라인`)와 수정 설계(어디를 어떻게)를 쉬운 말로 제시. 대안·더 단순한 방법이 있으면 함께.
- `status.md` 상태를 `분석완료`로 바꾸고 분석 내용을 상세히 기록(나중에 이 파일만 읽어도 재분석 없이 이어갈 수 있을 만큼).

### 7. 테스트 목록 초안 작성

`issue-start` 7단계와 동일하다.
- `$DOBBY_META/.issue-test/{이슈키}/test-plan/{이슈키}-test-plan.md`에 시나리오(S1, S2 …)를 self-contained하게 작성. 테스트 데이터는 DB 조회 도구가 있으면 실제 값을 찾아 채운다(조회만; 도구가 없으면 사용자에게 묻고, 그래도 없으면 가용 범위만). 환경은 `미정`으로 둔다. 상단 메타에 `작성 출처: agent-start` 기록.

## 오케스트레이션 메타 (상위 에픽 단위)

여러 하위이슈를 상위(에픽) 브랜치 위에 올려 서브 에이전트로 구현시킬 때, **메인 에이전트**가 유지하는 메타 계층. `agent-start`는 하위이슈별 착수·분석까지만 하고, 아래 메타는 **메인 에이전트가 P1(분석 검증·범위 배분) 단계에서 작성**한다. 이 문서는 그 규격을 정의한다.

### 디렉터리 레이아웃

```
$DOBBY_META/.agent-start/{에픽키}/
├── orchestration.md          # 메인 에이전트의 관제탑(보드)
├── agents/
│   ├── {에이전트슬러그}.md    # 서브/리뷰 에이전트별 계약(불변 컨트랙트)
│   └── ...
├── reviews/
│   └── round-{n}/
│       └── {에이전트슬러그}.md # 라운드별 리뷰 피드백
└── agent-logs.json            # {슬러그}→에이전트 대화 로그 경로(work-dobby가 수집)
```

- 계약 파일은 **불변 컨트랙트**, `status.md`는 **진행 상태**, `orchestration.md`는 **메인의 관제탑**으로 역할을 분리한다.
- 에이전트 슬러그는 파일시스템 안전하게 영문 kebab-case(예: `admin-agent`, `display-agent`, `review-agent`)로 하고, 파일 안 제목에는 한글 역할명을 함께 적는다.

### 서브 에이전트 계약 (`agents/{슬러그}.md`)

메인 에이전트가 범위 분석 후 작성한다. 각 서브 에이전트는 기동 시 이 파일을 **가장 먼저 읽고** 시작한다.

```markdown
# {역할명} 에이전트 계약

## 정체성
- 역할: {한 줄}
- 담당 이슈: {이슈키} (상위 {에픽키})

## 작업 환경
- 워크트리: $DOBBY_WORKSPACE/subtree/{repo}-{이슈키}
- 브랜치: {prefix}/{이슈키} (base: {상위 브랜치})
- status: $DOBBY_META/.issue-start/{이슈키}/status.md

## 수정 허용 범위 (충돌 방지 핵심)
- 허용: {파일/디렉터리 화이트리스트}
- 금지: {다른 에이전트 담당 영역}
- 경계/공용 파일: {있으면 명시 — 수정 시 메인에 먼저 보고}

## 사용 스킬: agent-impl

## 동작 순서
1. 이 계약 + status.md + 이슈 설명 숙지
2. agent-impl 실행(status 구현중 → 구현 섹션 기록 → 구현 → commit --no-verify → push)
3. 메인에 리뷰 요청 후 대기
4. reviews/round-{n}/{슬러그}.md 수신 → 허용 범위 내 수정 → 재푸시 → 재요청

## 주의 사항
- CLAUDE.md 컨벤션 준수 / 허용 범위 밖 수정 금지(필요 시 메인에 요청)
- 커밋 규칙: {feat|fix}: {이슈키} ... , --no-verify

## 완료 정의(DoD)
- 요구사항 구현 + status.md 구현 섹션 기록 + 커밋·푸시 + 리뷰 통과
```

### 리뷰 에이전트 계약 (`agents/review-agent.md`)

```markdown
# 코드리뷰 에이전트 계약

## 역할: 서브 에이전트 산출물 diff 리뷰
## 대상: {브랜치들} (각 base 대비 diff)
## 리뷰 기준
- 요구사항 충족(이슈/기획서 대비)
- correctness / CLAUDE.md 컨벤션 준수
- 범위 준수 검증: 각 에이전트가 계약의 허용 범위만 수정했는지
## 실행: 브랜치별 병렬 리뷰 가능
## 출력: reviews/round-{n}/{슬러그}.md (심각도·파일:라인·수정 제안)
## 통과 기준: blocking 이슈 0
```

### 오케스트레이션 보드 (`orchestration.md`)

```markdown
# {에픽키} 오케스트레이션 보드

## 에이전트 상태
| 에이전트 | 이슈 | 브랜치 | 상태(대기/구현중/리뷰중/완료) | 라운드 | 갱신 일시 |

## 범위 배분 (충돌 방지 결정)
| 영역/파일 | 담당 | 근거 |

## 실행 모드: 병렬 / 순차 (+ 사유)

## 충돌·경계 이력

## 이벤트 로그
- {YYYY-MM-DD HH:MM} {단계/사건} — {한 줄 설명} (예: `P4 어드민 구현중 진입`, `round-1 리뷰 완료: blocking 2`)
```

- **`갱신 일시`**: 그 에이전트의 상태가 마지막으로 바뀐 시각. "멈춤 vs 진행 중" 판단에 쓴다.
- **`## 이벤트 로그`**: 단계 전이·리뷰 결과·충돌 판단 등 사건을 **타임스탬프 1줄**로 누적한다(덮어쓰지 않는다). 나중에 이 로그만 훑어도 진행 궤적이 복원된다.

### 에이전트 상태 값 (정식)

피드백 루프·재통합을 포함한 실제 흐름을 반영한다. **서브 에이전트**와 **리뷰 에이전트**의 상태 집합이 다르다.

- **서브 에이전트**: `대기` → `분석중` → `구현중` → `리뷰중` → `수정중` → `재통합대기` → `완료`
  - `대기`: 시작 전 · `분석중`: agent-start 분석 중 · `구현중`: agent-impl 구현 중 · `리뷰중`: 리뷰 에이전트가 이 브랜치를 리뷰 중 · `수정중`: 리뷰/사용자/디자인 피드백 반영 중(라운드 n) · `재통합대기`: 수정·푸시 완료, 상위 브랜치 재병합 대기 · `완료`: 상위 브랜치 통합 완료
  - 피드백이 반복되면 `리뷰중` ↔ `수정중`을 오간다. 라운드 번호는 `라운드` 열에 적는다.
- **리뷰 에이전트**: `대기` → `진행중` → `완료`
- 상태가 바뀔 때마다 `갱신 일시`를 갱신하고 `이벤트 로그`에 1줄 남긴다.
- **보드 상태 칸에는 정식 값 하나만 적는다.** 라운드·사유 등 부가 정보는 `라운드` 열과 `이벤트 로그`에 적고, 상태 값에 괄호 주석을 붙이지 않는다.

#### 상태 전환 다이어그램 (서브 에이전트)

```
대기 ─▶ 분석중 ─▶ 구현중 ─▶ 리뷰중 ─┬─(blocking>0)─▶ 수정중 ─▶ 리뷰중  (라운드+1, 반복)
                                     └─(blocking=0)─▶ 재통합대기 ─▶ 완료
```

- `리뷰중 → 재통합대기`는 **blocking 0에서만**. `수정중` 이후에는 반드시 `리뷰중`(재리뷰)을 거친다(생략 금지).
- 리뷰 에이전트: `대기 ─▶ 진행중 ─▶ 완료` (라운드마다 반복).

### 메인 에이전트 책임 (요약)

1. 각 하위이슈에 `agent-start ... base={상위 브랜치}` 실행 → 착수·분석 확보.
2. 분석이 기획서·Jira·(있으면)Figma와 맞는지 교차 검증.
3. **구현에 필요한 코드를 먼저 분석해 에이전트별 수정 범위가 겹치지 않도록 화이트리스트를 정하고** `orchestration.md`·`agents/*.md`에 기록. 겹침이 불가피하면 **병렬 대신 순차 실행**으로 결정하고 사유를 남긴다.
4. 서브 에이전트 기동(계약 파일 경로를 첫 지시로 전달) → 리뷰 에이전트로 코드리뷰 → 피드백 루프(최대 2~3라운드, 이후 사용자 에스컬레이션) → 리뷰 클린 시 상위 브랜치로 순차 머지·푸시.

## status.md (확장)

- 경로·규격은 `issue-start`의 "작업 상태 파일"과 동일하되, **상태 값에 `구현중`·`리뷰중`·`수정중`을 추가**한다.
  - `착수`(agent-start 워크트리 생성) → `분석완료`(원인·수정 설계) → `구현중`(agent-impl 착수, `## 구현` 섹션 기록) → `리뷰중`(리뷰 대기·진행) → `수정중`(피드백 반영, 라운드 반복 시 `리뷰중`↔`수정중`) → `해결됨`(issue-test 실패 0·중단 0 마감).
- `작성 출처`에 `agent-start`를 쓴다.

## 주의

- **사실 기반 원칙(최우선)**: 모든 분석·판단은 직접 확인한 사실(코드·문서·실행/조회 결과)에만 근거한다. **추측하지 않는다.** 확인 못 한 것은 "미확인"으로 명시한다. signature·이름만 보고 동작을 단정하지 않는다.
- 어려운 용어를 지양하고 쉬운 말로 설명한다.
- Jira 상태 전환은 `statusCategory`로 판정한다. 종료된 이슈·애매한 전이는 사용자에게 확인한다.
- 커밋·푸시는 하지 않는다(구현은 `agent-impl`, 최종 머지·푸시는 메인 에이전트가 사용자 동의 후).
- 같은 이름의 워크트리·브랜치가 있으면 자동 생성하지 말고 backfill 규칙으로 처리하거나 사용자에게 확인한다.
- 워크트리에는 `node_modules`가 없어 pre-commit 훅이 실패할 수 있으므로, 이후 커밋은 `--no-verify`로 훅을 건너뛴다.
